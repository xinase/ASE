## 11.1 什么样的软件开发流程、开发模式才是最适合我的？

学生原以为软件工程课会教如何从手工代码作坊进化到标准化工厂，但第11章只有短短数十页，感到失望，询问什么样的流程/模式最适合自己。

**问题的核心：** 没有“最适合”的通用流程，只有“最匹配当前团队、项目、风险”的流程。

**回答：**

别指望一本书或者一章给你一个“最佳实践”的银弹。第11章讲的“白板优先”、ADR、建模方法，乃至整个自动编程历史，都在说明一个道理：**流程和模式是工具，不是信仰。**

最适合你的流程，取决于三个变量：
1. **团队规模**：一个人随便搞；三五个人，白板+每日站会+Git Flow就够了；几十人，才需要Jira、CI/CD门禁。
2. **需求确定性**：客户需求像石头一样硬？瀑布也未尝不可。需求像水一样天天变？那就必须敏捷、短迭代。
3. **风险等级**：银行核心系统，需要形式化方法、严格ADR；一个课程项目，白板草图+有记录的关键决策已经足够。

书里11.3.5节的“苦涩的教训”也适用于流程选择：**别试图用一套复杂规则管住一切，拥抱简单、可扩展、能随着反馈调整的流程。**

**扩散思考：**

你感觉“只有数十页”，恰恰是因为**设计实现的核心不在于厚文档，而在于把事情说清楚、把决策记下来、把代码跑起来**。第11章真正有价值的是：
- **白板优先**：画不清楚就别写代码。
- **ADR**：把“为什么”留下来，否则你就是在重复踩前人踩过的坑。
- **CI/CD + 脚手架**：让机器帮你做验证，而不是靠人开会。

从“手工作坊”到“工业化”，不是靠读一本秘籍，而是靠**持续引入自动化门禁、代码审查、单元测试覆盖率、每日构建**这些具体实践。AI时代，这些实践不是变多余了，而是更重要了——因为AI生成的代码需要更强的安全网。

## 11.2 设计文档如何平衡严谨性与复杂性？

学生询问设计文档如何在严谨与复杂之间平衡，以及实际项目中是否使用JML等规格语言，如何保证规格的正确性。

**问题的核心：** 严谨 ≠ 复杂。用“足够的严谨”解决“真正的风险”，而不是用形式化解决一切。

**回答：**

书里11.1.1节的“白板优先”已经给了答案：**简约文档 + 关键决策记录（ADR）**。

- **不需要所有细节都形式化**。像JML这类规格语言，在航天、高铁等安全攸关领域有用，但普通商业软件用起来代价太高。绝大多数项目，用自然语言 + 清晰的例子 + 边界条件描述，配合单元测试作为“可执行规格”，就足够了。
- **严谨性体现在“关键之处”**：对外接口的契约、并发模型、错误处理策略、数据一致性边界。这些地方用表格、约束条件、甚至简单的断言来明确。至于内部实现细节，让代码和测试自己说话。

**如何保证规格的正确性？**
- **复审**：让另一个有脑子的人看。
- **对照测试**：写出规格的同时，写出对应的测试用例。如果测试能跑通且覆盖边界，规格大概率是对的。
- **使用AI反向检查**：把你的规格文档喂给AI，让它生成疑问列表。那些它觉得模糊的地方，就是你需要加固的地方。

**扩散思考：**

书里11.2.5节提到的形式化方法（TLA+）在处理分布式系统的一致性时非常强大，但学习曲线陡峭。一个务实的做法：**只对系统中“一旦出错就会很惨”的核心模块使用形式化或半形式化描述**，其余部分信任代码审查和测试。不要为了“看起来专业”而引入不必要的复杂性——那正是CASE工具失败的原因之一。

## 11.3 在软件开发中，如何应用建模方法？

学生使用过ERD、DFD、UML，但经常出现“有了模型却不知道下一步做什么”或者模型与实现脱节的情况，询问如何避免“为建模而建模”。

**问题的核心：** 建模的目的是沟通与决策，不是产出漂亮图片。建模后给谁看？给“需要理解系统并据此行动的人”看。

**回答：**

你问“建模后给谁看？”——这个问题才是真正的钥匙。

- **给自己看**：画个草图理清思路。别用UML，用白板加手机拍照。够用了。
- **给队友看**：强调关键结构和交互，忽略无关细节。比如画时序图只画核心消息，别把每个if-else都画进去。
- **给未来维护者看**：用ADR记录决策原因，用模块关系图（不用标准UML，用方框+箭头）描述依赖。
- **给AI看**：把图用文本形式（如Mermaid）记录下来，AI可以帮你生成代码骨架或测试。

**怎么避免脱节？**
书里11.2.4节引用了Hans-Peter Hoffmann的例子：客户有了模型却不知道下一步做什么。原因很简单——**模型没有和生产代码的“任务”挂钩**。正确做法：
1. 从模型中的一个模块/实体出发。
2. 写出它的接口定义（可以是伪代码或TypeScript interface）。
3. 写出它的核心测试用例。
4. 然后才写实现。

模型只是地图，代码才是脚步。地图画得再漂亮，不迈腿就是废纸。

**扩散思考：**

AI时代，建模的价值反而上升了。因为AI可以快速根据你的模型图（Mermaid、PlantUML）生成代码骨架和测试。但前提是：**你的模型要足够清晰、边界足够明确**。所以，与其花时间画完美UML，不如花时间把模块的输入、输出、副作用、错误处理用自然语言+示例写清楚。这才是AI能消费的“规格”。

## 11.4 在看到初始效果和了解实现细节后，是否可以直接开始写代码？

学生认为很多场景下（需求简单、或需要尝试优化）直接写代码更高效，设计文档显得“马后炮”，质疑写文档的必要性。

**问题的核心：** 设计文档不是为了“流程正确”，而是为了**防止你在复杂场景下犯低级错误，并留下决策痕迹**。

**回答：**

你说得对：简单的一次性脚本、需要快速试错的原型，直接写代码没问题。但那是**编程**，不是**软件工程**。

书里11.4节（实际上是11.3之后的内容）强调的是：当你面对的是一个会持续演进、多人协作、需要长期维护的系统时，**没有设计文档就像没有地图的远航**。

你举的例子——抓取新闻列表，方案从页面解析改成AJAX请求——恰恰证明了设计文档的必要性。如果当时有一页纸记录了：
- 方案A（页面解析）的预期性能、风险
- 方案B（直接请求）的前提条件
那么你就不需要“写代码测试才发现问题”，而是在设计阶段就可以快速排除A。

**写文档的正确姿势不是“先写完美文档再写代码”，而是：**
1. 先写一页以内的“设计草稿”（白板图+关键决策）
2. 实现核心原型验证
3. 根据验证结果更新设计文档（只更新关键部分）
4. 最后作为“活文档”留存

这样文档不是马后炮，而是**决策的固化**。

**扩散思考：**

AI时代，你可以让AI根据你的代码反向生成设计文档初稿，再手动修正。这大大降低了写文档的成本。但**决策本身的责任**——为什么选A不选B——仍然需要你来记录。否则三个月后，AI也解释不了当初的权衡。

## 11.5 如何在开发日常管理中包容学生的不同点（并行任务、时间不统一）？

学生认为专业开发者串行专注项目，而学生同时应付多门课程、临时任务，导致很难找到共同会议时间，询问如何设计日常管理以包容这些不同。

**问题的核心：** 用异步协作和文档驱动取代同步会议和口头约定。

**回答：**

你提出的“以文档为核心的日常管理机制”是对的。具体怎么做：

1. **拒绝“开会讨论”，拥抱“异步决策”**：使用GitHub Issues、Notion、飞书文档。任何人提出一个问题，大家可以在24小时内评论。PM每天花30分钟汇总。
2. **固定但稀疏的同步点**：每周一次15分钟站会（可以用腾讯会议/微信群语音），只做三件事：每个人说“我上周完成了什么、下周计划什么、有什么阻塞”。其余时间各干各的。
3. **代码审查代替口头沟通**：所有代码变更必须通过Pull Request，在PR里讨论实现细节。这样时间不同步也能协作。
4. **使用AI当“会议秘书”**：如果必须开会，录屏+AI自动生成纪要和待办，发到群里。缺席的人自己看。

书里11.7节强调的源代码管理和CI/CD，在学生项目中同样适用。**把一切都变成文本、代码、可追溯的记录**，就能打破时间枷锁。

**扩散思考：**

学生团队的“并行任务”其实是未来职场常态——程序员同时参与多个项目、处理线上故障、应对紧急需求。所以现在学会异步协作，反而是优势。关键是**不要让学生PM变成“催债人”，而要变成“流程设计者”**：设计一套让每个人即使在不同时间也能高效贡献的规则。

## 11.6 关于多人合作中的代码如何同步？

学生发现多人修改同一代码库时merge冲突频繁，即使限制了签入时间段也难解决，询问如何减少冲突。

**问题的核心：** 冲突的本质是**模块划分不清晰 + 集成频率过低**，工具只能缓解，不能根治。

**回答：**

你提到“自动构建门禁”只解决了“构建不失败”，没解决冲突本身。书里11.7节其实已经给出了方向：

1. **模块解耦**：如果两个人频繁改同一个文件，说明模块划分太粗。重构！按功能拆分成独立文件/包，定义清晰接口。
2. **高频集成**：每天至少拉取一次主分支（git pull --rebase），而不是一周才merge一次。冲突越早处理越简单。
3. **特性分支短命**：一个分支的生命周期不超过两天。长分支是大规模冲突的温床。
4. **使用git rerere**：让git记住你解决冲突的方式，下次自动处理。

**最实用的建议**：
- 团队成员在修改前先git pull --rebase
- 推送前再次git pull --rebase
- 如果出现冲突，不要慌，用IDE的可视化merge工具（VSCode、IntelliJ都很好）
- 实在搞不定，喊上冲突对方，两个人一起看十分钟就解决了

**扩散思考：**

AI可以帮你分析冲突并给出合并建议（比如GitHub Copilot或Cursor）。但AI无法替你决定“这个功能应该放在哪个模块”。所以，**冲突的根本解还是在设计阶段**：好的模块边界和接口定义，能让两个开发者各自独立修改不同部分，永远碰不到一起。这正是11.1节“白板优先”和ADR的价值所在。

## 11.7 “小强地狱”这种按照小强的数量判断是否需要修复bug的评判标准是否合理？

学生认为数量不是唯一标准，有些严重bug即使只有一个也应立即修复，而低优先级的bug可以积攒集中处理，质疑单纯按数量阈值决定是否入狱的合理性。

**问题的核心：** “小强地狱”是一种**行为引导机制**，不是精确的缺陷管理工具。其核心目标是防止个人累积过多bug拖垮团队。

**回答：**

你指出的问题是对的：**一个导致系统崩溃的bug和 10 个界面错别字，危害完全不同**。所以实践中，“小强地狱”的阈值通常只计算**中高优先级**以上的bug（比如严重性为“严重”或“一般”），而“轻微”和“建议”类不计入。另外，一个程序员负责的代码可以搞出 10 个界面的错别字，而没有马上改掉，或者用 AI工具自动修复，这说明了什么深层的问题？

但“小强地狱”的真正价值不在于精确度量，而在于：
- **打破“先开发新功能，以后修bug”的拖延心理**。没有阈值，很多人会把bug无限期搁置。
- **强制团队暴露质量债务**。当连续多人入狱，说明测试或设计流程出了问题，需要改进。

书中也说了：入狱人数比例应控制在5%-30%，若太高则要调整阈值或计算方式。所以它不是死板的。

**扩散思考：**

一个更现代的做法：**在CI门禁中设定“bug上限”**。比如，当某个模块的未修复中高优先级bug超过3个，该模块的所有新PR自动被门禁拒绝，直到bug数降下来。这比“小强地狱”更自动化，也更公平。但无论哪种，本质都是**让质量债可见、可管理**。

## 11.8 “小强地狱”中的阈值如何界定？

学生询问如何具体设定bug数量的阈值，以及应该根据团队能力还是bug的影响程度来划分。

**问题的核心：** 阈值是动态的，由**历史数据 + 团队能力 + 业务容忍度**共同决定，没有魔法数字。

**回答：**

书里已经给了一个实用原则：**入狱人数比例控制在5%-30%**。这意味着：
- 如果团队10人，阈值的设定应该使得任何时候最多有1-3人同时入狱。
- 如果发现经常没人入狱，说明阈值太宽松（或者bug太少，那就恭喜）。
- 如果经常超过3人入狱，说明阈值太严格，或者团队整体质量太差，需要调整阈值或改进流程。

**具体操作步骤：**
1. 统计过去2周每人被分配的中高优先级bug数量分布（平均值、标准差）。
2. 初设阈值 = 平均值 + 1.5倍标准差。
3. 运行一周，观察入狱人数比例。若太高则调高阈值；若太低且质量没问题，可调低阈值倒逼改进。
4. 阈值宣布后，至少稳定一到两周再调整，避免朝令夕改。

**另外要注意**：阈值应当区分bug的严重等级。通常只统计“严重”和“一般”级别。“轻微”或“建议”类要么不计入，要么权重设为0.2。

**扩散思考：**

AI可以帮你自动计算每个开发者的bug关闭率、平均修复时间，甚至预测“按当前趋势，某人将在几天后入狱”。但阈值本身仍然是管理决策。一个小团队更简单的做法：**每周例会上公开每个人的未修复bug数（按严重程度分级），让团队自己觉得“有点丢人”而主动修复**。社会压力比阈值更有效。

## 11.9 我们是否需要一个AI时代的脚手架清单？

学生担心AI生成代码激增而测试脚手架跟不上，导致质量崩溃；同时担心AI生成的技术债务更难识别。询问AI时代需要什么样的工具、流程、规范，以及如何平衡快速迭代与稳健工程。

**问题的核心：** AI时代，脚手架不再是“限制”，而是“安全网”。需要一套以**自动化验证**和**架构约束**为核心的清单。

**回答：**

书里11.5.5节已经给出了方向：“驾驭工程”——不是靠流程限制人，而是靠脚手架规范AI。

**AI时代的脚手架清单（至少包含以下五项）：**

1. **单元测试覆盖率门禁（强制）**：AI生成的代码必须同步生成单元测试。CI检查覆盖率，低于阈值（如80%）禁止合并。这是防止AI“幻觉”的第一道防线。

2. **架构决策记录（ADR）驱动的约束**：重要架构决策（如模块边界、依赖方向、并发模型）写成ADR，然后用自动化工具（如ArchUnit）检查代码是否违反这些约束。AI生成的代码如果绕过架构，CI直接拒绝。

3. **静态分析 + AI审计**：除了传统的Lint，使用AI工具对PR进行“架构一致性”和“安全漏洞”的二次审查。AI可以标记出“看起来合理但实际有隐患”的代码模式。

4. **技术债务追踪**：每个AI辅助生成的功能模块，必须显式评估“预估技术债务”（比如从1到5级）。当债务等级超过阈值，团队必须安排专门的“还债迭代”。

5. **可观测性作为质量反馈**：利用AI分析日志、追踪和Metrics，自动识别哪些模块频繁出问题、哪些AI生成的代码导致线上故障。把这些数据反向输入到代码审查门禁中。

**如何平衡快速迭代与稳健工程？**
- **快速迭代**：靠AI生成原型和常规功能。
- **稳健工程**：靠上述脚手架确保每一行进入主干的代码都经过验证。不要因为“快”而跳过门禁。

**扩散思考：**

书里11.3节的自动编程历史告诉我们：每一次技术飞跃都会带来新的“偶然复杂性”。AI代码生成降低了“写代码”的成本，却放大了“验证代码”的难度。未来的工程师不是“写代码更快的人”，而是**设计脚手架、训练AI遵循规范、快速识别AI生成债务的人**。这才是11.6节“源代码管理”在AI时代的真正含义——管理的不再只是人工代码，而是人与AI的联合产出。
