# 中关村学院 · 现代软件工程课程计划（2026）

> 本课程全部内容将在 GitHub 开源，欢迎外校同学 Star / Fork / Issue / PR 参与学习。


## 第一部分：课程概要（快速了解版）


### 一、课程定位与先修路径

本课程处于中关村学院人工智能课程体系的**第三阶段**，面向已完成前序学习的博士一年级学生：

- **第一站 · 人工智能通识课**：Python 编程入门，理解"程序 = 算法 + 数据结构"
- **第二站 · Vibe Coding 系列 / 智能体工程**：了解个人开发规律，理解"软件 = 程序 + 软件工程"（特别是 AI 时代的驾驭工程）
- **第三站 · 现代软件工程（本课程）** ：深入理解**团队层面**的规律，覆盖 OPC 中 Company 如何创新 —— "软件 = 程序 + 软件工程"；"软件企业 = 软件 + 商业模式"

本课程面向博士一年级学生（含大量跨专业 AI 学习者），以"中关村学院文创"为主赛道（也欢迎其他赛道），在 14 周内完成从结对设计到团队开发、从 V1 到 V3、从内部评审到外部投资的完整软件工程实战。课程以"技术发布会"收官，优秀团队将获得风险资本投资及学院各相关部门的孵化支持。

### 二、核心理念：船舱 · 甲板 · 山巅

> **"AI 都能写代码了，编程和软件工程课还有价值吗？"**

这正是本课程要回答的问题。人工智能正在彻底改变软件开发的格局，CoPilot 这样的工具已经成为标配，但我们也看到了新的问题：不少学生因为基础不牢，在实际项目中陷入了"氛围编程"的困境 —— 过度依赖 AI 生成代码，不知道背后的原理，更不知道怎么构建高质量的软件。

本课程以"**以终为始**"的理念进行"**干中学**"教学。这个"终"不是考试分数，而是真实工程和科研的挑战。课程中的教学设计，可以看作是从"**船舱、甲板、山巅**"三个角度出发，系统培养学生的综合能力：

- **船舱中的水手**：像船舱水手一样埋头实干，深入具体项目实践，亲身经历从代码编写、调试纠错到团队沟通、版本交付的全流程压力与挑战
- **甲板协作者与竞争者**：在评价与互动中思辨，跳出技术执行的单一视角，在观察、比较、协作与批判中拓展认知边界
- **山巅观察者与思考者**：进行大胆的俯瞰与思考，不时超越当下项目的具体细节，以更长远的视角思考软件工程学科的本质，通过 OPC 等形式证明自己的创新点子 + 高质量工程 + 合适的商业模式 = 初步成功。

通过这三重视角的转换，学生得以全程参与项目实践，经历从需求分析到部署运维的完整软件生命周期，最终锤炼出在 AI 辅助下进行工程决策与创新的核心判断力。


### 三、核心原则

- **面向实战**：做真正的软硬件结合项目，解决真实用户的真实问题
- **结果导向**：不规定何时写什么代码，以里程碑时刻"可演示的产出"为衡量标准
- **开源文化**：全部课程资料在 GitHub 开源，欢迎外校同学参与学习
- **学习闭环**：从学期初提出问题，到学期末回答自己的问题，完成完整的认知循环
- **AI 工具深度使用**：贯穿创意、设计、编码、测试全流程


### 四、课程整体逻辑（14 周）

- **第 1 周**：课程总览 + 快速阅读《构建之法》+ 闪电配对
- **第 2–3 周**：结对项目（设计驱动，PPT/原型图，不写代码）+ 热身评审
- **第 4 周**：结对合并成团队（2+2=4，≥3 人 OPC 达标）
- **第 5–6 周**：团队 V1 冲刺 + 里程碑检查 #1（天使投资人介入）
- **第 7–8 周**：人员流动 + V2 冲刺 + 里程碑检查 #2（外部资本 + 本院孵化部门介入）
- **第 9–10 周**：V3 冲刺 + 技术发布会（四方联合评审 + 现场签单）
- **第 11–12 周**：硬件打样等待期 / 优秀项目深度孵化 / 补强辅导
- **第 13 周**：学习闭环总结会——回答学期初的 n 个问题 + 课程复盘
- **第 14 周**：缓冲（个别辅导、资料归档）


### 五、关键里程碑与外部参与节点

- **热身赛 · 结对评审（第 3 周）** ：评审团为主讲老师 + 工程师 + 助教；核心关注痛点的真实性、方案的说服力、设计的完整性
- **里程碑 #1 · V1 发布（第 6 周）** ：评审团增加天使投资人（外部）；核心关注软硬件是否连通、核心场景是否跑通
- **里程碑 #2 · V2 发布（第 8 周）** ：评审团增加外部资本 + 孵化部门；核心关注 V1 反馈是否被采纳、产品稳定性、品牌潜力
- **里程碑 #3 · 技术发布会（第 10 周）** ：评审团多方联合；核心关注技术 + 文创 + 商业 + AI 创新


### 六、学习闭环

- **学期初（第 1 周）** ：快速阅读《构建之法》核心章节，每位学生提出至少 5 个问题
- **学期中（第 2–12 周）** ：通过课堂、项目、答疑、评审持续寻找答案
- **学期末（第 13 周）** ：学习闭环总结会，逐一回答自己的问题，提交"学习闭环报告"


### 七、课程产出

- 每个团队完成一个可演示的软硬件产品原型（至少达到"功能原型"级别）
- 所有代码、设计图、演示视频在 GitHub 公开
- 每位学生完成从提问到回答的"学习闭环报告"
- 优秀团队获得学院采购订单、外部资本跟投意向、产业化部门的支持，等。


### 八、外校学生参与方式

## 第二部分：详细周计划


### 团队分工

- **主讲老师**：核心概念讲授、主持评审、对接院方及外部资源
- **工程师（硬件 / 产品设计）** ：负责文创硬件选型、通信协议、3D 打印设备维护与建模指导、打样辅导、硬件 BOM 清单及电路图开源；同时提供基础 3D 打印模板库（底座、支架、Logo 立体字等），在产品外观与结构设计上提供轻量咨询。**3D 打印相关内容定位为可选赋能资源**，服务于"产品感"提升，不额外增加学生课业负担。
- **工程师（软件 / AI）** ：代码规范、API 设计、AI 工具应用，示例代码及 API 模板开源
- **助教**：流程管理、站会追踪、学习进度维护、维护 GitHub Issues 及 PR 审核
- **外校学员（围观参与）** ：通过 GitHub 参与讨论、提交 Issue 提问、作为"大众评审团"参与评分


### 第一阶段：快速阅读 + 闪电配对 + 结对项目（第 1–3 周）

> **对应"船舱"视角**：学生如投身船舱的水手，深入具体项目实践，从零开始构建产品，亲身体验从创意到设计的全过程。这一阶段也是学习闭环的起点，学生通过快速阅读和提问，在教师的引导下完成"从疑问到思考"的初步转变，将"我不知道我不知道"的模糊领域概念，转化为"我知道我不知道"的清晰问题清单。


**第 1 周 · 课 1：课程总览 & "从提问开始"的动员**

- 前半（主讲老师）：发布"中关村学院文创"主赛道（含硬件要求），介绍课程全部内容将在 GitHub 开源，欢迎外校参与；介绍其他备选赛道。同时引入"船舱·甲板·山巅"三重视角，说明学生在课程中将以三种身份去实践
- 后半（主讲老师）：讲解"快速阅读与提问"方法论——"Just do it"原则，说明即使在不完全理解的情况下，集中提问也能为大脑做好准备；展示去年优秀提问案例，点明"把不懂的地方有条理地描写出来"是核心要求
- 关键产出：学生注册 GitHub 账号、Star 仓库；发布作业"快速阅读《构建之法》核心章节（第 1、2、4、6、8、9、13、16 章），提出至少 5 个你还不知道答案的问题"，发布个人博客；课程 README 发布，外校学员可开始关注


**第 1 周 · 课 2：闪电配对工作坊 & 阅读启动**

- 前 15 分钟（主讲老师）：讲解配对规则——围成内外两圈，每 3 分钟轮转一次，双方互记技术特长、兴趣方向、性格特点、最想做的产品。鼓励学生交流彼此对《构建之法》的初步印象和感兴趣的方向
- 中间 60 分钟（全场）：闪电配对，每人至少聊 8–10 人，记录"心动合伙人"榜单
- 最后 15 分钟（助教统计）：现场公布配对结果（2 人一组），未配对者由助教协调
- 关键产出：结对名单确定；每组提交《合伙人档案》；配对规则视频录播开源；助教收集所有提问并归档到 GitHub 仓库


**第 2 周 · 课 3：NABCD 模型 & 创意激发**

- 前半（主讲老师）：深度讲解 NABCD，以"中关村学院文创"为例做示范。课中设置"提问-解答"环节，选取几份学生提交的关于"需求""创新"等方面的优秀提问，组织全班讨论并现场答疑
- 后半（工程师 · 硬件/产品设计）：展示 AI 设计工具（Midjourney / Figma AI），快速生成产品外观；工程师同步演示一个极简的 3D 打印工作流——如何把一张设计草图用 TinkerCAD 在 10 分钟内拉出一个可打印的简易粗模，并启动打印。**此环节为可选赋能**，不要求学生立即动手，只为展示"从图到物"的可行性
- 关键产出：每组产出 3 个创意方向；创意库开源（外校可提交 Issue 提供创意）；工程师发布 3D 打印入门视频和模板文件库（含简易底座、支架、Logo 立体字等）至 GitHub


**第 2 周 · 课 4：结对项目启动 & 设计驱动**

- 前半（工程师 · 软件/AI）：讲解"设计驱动开发"——PPT / 设计图要做到什么程度才算"可评审"
- 后半（实战）：各组现场做 PPT / 原型图，工程师巡场指导。助教整理全班提问形成"疑问热力图"，用于指导后续答疑课的重点讲解，并发布在 GitHub 供外校同学讨论。硬件/产品设计工程师在巡场时向有意的团队介绍：如果想在评审时手上有一个"桌上信物"，可以如何快速建模并打印（提供模板，30 分钟工作量，完全可选）
- 关键产出：每组确定 1 个方向，开始做设计；设计模板开源


**第 3 周 · 课 5：场景设计 & 用户体验**

- 前半（主讲老师）：基于场景的设计方法，强调文创产品的"开箱体验"和"用户旅程"
- 后半（工程师 · 硬件/产品设计）：硬件选型如何融入场景（NFC 碰一碰、蓝牙连接等），同时补充：如果产品有固定的外形设想，可以用 3D 打印快速制作一个"尺寸占位模型"放在桌上，帮助团队在场景推演时更直观地感受产品大小、握持感与摆放方式——**全程只做演示，不做强制要求**
- 关键产出：各组完善设计图中的"用户使用场景"部分；场景设计案例开源


**第 3 周 · 课 6：结对项目评审会（热身赛）**

- 全程（主讲老师主持 + 工程师 / 助教当"天使投资人"）：每组 6 分钟演示 PPT / 设计图，重点考察痛点是否真实、解决方案是否有说服力、设计是否完整、硬件选型是否合理，当场投虚拟币 / 评分。**加分项（非必须）** ：如有团队选择打印了简易粗模放在桌上展示，评审团可给予"产品感"印象加分（体现在口头点评中，不折算具体分数）
- 关键产出：结对学生项目评分 + 排名；优秀设计进入"待合并池"；评审录像开源（脱敏）


### 第二阶段：结对 → 团队合并（第 4 周）

> **对应"甲板"视角的萌芽**：学生开始跳出自己的小组，看到其他团队的设计与思路；通过"千帆竞发图"和公开仓库，所有团队的进度相互可见。同时，助教在此阶段发布"疑问热力图"，帮助学生看到他人的思考角度，拓展认知边界。


**第 4 周 · 课 7：团队合并 & 角色分配**

- 前 30 分钟（主讲老师）：宣布合并规则——鼓励方向相近的结对组"联姻"（2+2=4 人），落单者可加入任意团队，每个团队必须 ≥3 人（OPC 达标）；讲解团队角色（PM / Dev / HW / Test）及其职责。新增角色建议：团队可视自身需要决定是否设立"产品设计"角色，负责对接 3D 打印相关事务（可选，可兼任）
- 中间 40 分钟（助教协调）：各组谈判、合并、认领角色
- 最后 20 分钟：各新团队提交《团队章程》和《合并声明》
- 关键产出：团队名单确定（≥3 人）；团队章程模板开源


**第 4 周 · 课 8：从设计图到系统架构**

- 前半（工程师 · 软件/AI）：如何把 PPT / 设计图转化为系统架构图（前端 / 后端 / 硬件 / 数据流）
- 后半（实战）：各组现场画架构图，工程师巡场指导。硬件/产品设计工程师穿插说明：如果产品有外壳或结构件需求，架构图可同步标注"物理结构层"（如外壳尺寸、按键开孔位置、散热空间），但这**只作为设计文档的可选补充内容，非必要产出**
- 关键产出：各组提交《系统架构设计》初稿；架构图模板开源


### 第三阶段：团队 V1 冲刺（第 5–6 周）

> **本阶段核心目标**：各组根据自身能力自主推进开发，在第 6 周里程碑检查时能演示一个可运行的端到端核心场景。
> 
> **对应"船舱"视角的深化**：学生深入代码和硬件细节，亲身经历从零到一的构建过程。面对真实项目中的各种"肮脏细节"——环境配置、依赖冲突、硬件不响应、合并冲突——学生必须在具体实践中体会软件工程的"代价"，理解为什么需要各种工程实践和工具。


**第 5 周 · 课 9：团队冲刺启动 · 目标对齐与任务认领**

- 前 30 分钟（主讲老师）：发布 V1 冲刺的核心目标和检查点要求。强调本阶段唯一的评判标准是"第 6 周能演示一个可运行的核心场景"，不限制具体编码方式和时间。展示上届优秀 V1 Demo 视频
- 中间 40 分钟（助教协调 + 工程师巡场）：各组自行完成以下工作——将结对项目的设计图/PPT 转化为具体的功能清单；自行决定任务拆分方式和认领（可使用 GitHub Projects、Trello 或实体白板）；确认团队协作工具
- 最后 20 分钟：各组向助教口头汇报本阶段计划。硬件/产品设计工程师可在巡场时提示：如有团队希望在 V1 演示时让硬件"立起来、看起来更规整"，实验室的 3D 打印机可免费使用，基础底座模板在 GitHub 可直接下载打印，耗时约 1 小时。**完全按需自取**
- **本周衡量产出**：各组提交一份《V1 冲刺计划》（包含目标功能清单、成员分工、协作方式），不考核代码量


**第 5 周 · 课 10：硬件开发环境搭建 & 技术选型确认**

- 前半（工程师 · 硬件/产品设计 + 工程师 · 软件/AI 联合）：开放硬件工作台和软件调试环境，工程师提供技术选型咨询，但不替学生做决定
- 后半（工程师巡场 + 助教记录）：各组根据自身能力自主搭建开发环境，工程师巡回解答技术问题。鼓励能力强的组直接开始编码，基础弱的组先跑通"Hello World"级示例。硬件/产品设计工程师在实验室一角设置"3D 打印快闪摊位"，向感兴趣的团队展示：如何测量开发板孔距、如何在 Fusion 360 中快速拉出一个固定底座、如何切片打印。全程 15 分钟演示，**不占用正式课时，自愿参加**
- **本周衡量产出**：每组至少有一台开发机能成功编译并运行一个最小示例程序（如 LED 闪烁、屏幕显示文字、或一个"Hello World"网页），证明开发环境已就绪


**第 6 周 · 课 11：自由冲刺日（一）**

- 全程（工程师 + 助教巡场，主讲老师观察）：纯实战冲刺时间。各组根据自身计划推进开发，工程师提供技术指导。助教记录各组进展作为过程性档案
- 鼓励能力强的组进行团队内结对编程，基础弱的组可以集中攻克核心模块。3D 打印机持续开放，有需要的团队可自行切片、排队打印，助教协助管理打印队列
- **本周衡量产出**：无固定产出，但助教会记录各组进展。目标是让各组在课 12 前完成核心功能的至少 60%


**第 6 周 · 课 12：里程碑检查 · V1 核心场景演示**

- 全程（主讲老师主持 + 工程师 / 助教 / 天使投资人到场）：每组进行 10 分钟现场演示（可含代码走查、硬件操作、界面交互），重点展示——核心场景是否跑通（如：用户打开 APP→选择文创产品→碰一碰 NFC→看到数字内容）；硬件是否可运转（如：LED 亮、电机转、屏幕显示）；界面是否可交互（如：点击按钮有反馈、页面跳转正常）
- 评委不打断演示，结束后提问 5 分钟。重点关注"演示的完整性和可信度"，而非代码行数或技术复杂度。**如团队使用了 3D 打印底座或简易外壳使硬件呈现更规整**，天使投资人会在点评中给予"产品感"正向反馈，作为软性加分
- 天使投资人根据演示效果给出"种子轮虚拟投资"（评分），并给出改进建议
- **本周衡量产出**：各组必须完成一个端到端的核心场景演示（即便只是模拟数据或硬编码），证明"产品逻辑走得通"
- **关键底线**：若演示完全失败（如无法开机、界面崩溃、逻辑不通），该组需在课后一周内补录视频重新提交，否则 V1 阶段记为"不通过"


### 第四阶段：V1 反馈 + 团队重组与 V2 冲刺（第 7–8 周）

> **本阶段核心目标**：根据 V1 反馈进行人员重组，各组自主推进 V2 改进，在第 8 周里程碑检查时演示一个比 V1 更稳定、更完整的版本。
> 
> **对应"甲板"视角的深化**：这是课程中"甲板协作者"视角最鲜明的阶段。通过强制人员流动，学生必须跳出自己团队的舒适区，去理解、评判、修改他人的代码。这一过程模拟真实企业和动荡的创业团队中因人员流动、项目重组带来的交接挑战，让学生在"疼痛"中深刻体会"可维护性"和"知识显性化"的珍贵。正如往届学生所言——"当我被迫去接手别人的代码，或者看着别人把我的代码改得面目全非时，我才痛彻心扉地理解了什么是'可维护性'。"博客和代码全网公开 + 千帆竞发图动态展示进度，让认知边界进一步拓展。


**第 7 周 · 课 13：V1 回顾与团队重组**

- 前 40 分钟（主讲老师 + 工程师 · 软件/AI）：不进行代码走查，改为"产品回顾"。主讲老师逐一点评各组 V1 演示的亮点与不足（侧重点：场景完整性、用户体验、技术可行性），工程师补充技术改进建议
- 后 50 分钟（主讲老师 + 助教协调）：根据 V1 表现和团队意愿，宣布人员流动规则。流动依据是"产品缺口"（如：某组缺硬件能力，某组缺前端能力），而非"谁代码写得差"。各组自行谈判、吸收新成员，助教协调。硬件/产品设计工程师可在此环节提供一个额外的信息供团队参考——如有团队希望在产品外观上有突破，可以吸纳有 3D 建模基础的同学作为兼任设计角色
- **本周衡量产出**：各组提交新的《团队名单》和《V2 改进计划》（重点说明：V1 演示暴露了什么问题，V2 如何改进），不考核代码


**第 7 周 · 课 14：自由冲刺日（二）**

- 全程（工程师 + 助教巡场）：纯实战冲刺。新组员融入团队，工程师提供针对性指导。各组根据《V2 改进计划》自主推进开发
- 鼓励能力强的小组重构代码、增加测试；基础弱的小组专注于修复 V1 演示中暴露的 bug。同时鼓励团队成员在协作中保持开放心态——好的工程师要能说出代码的坏味道，也要能接受别人挑刺。硬件/产品设计工程师在当天中午开放 1 小时"设计诊所"——团队可以带着建模问题来咨询，或请工程师帮助检查模型是否适合打印。**完全自愿预约**
- **本周衡量产出**：各组在课 16 前完成 V2 核心改进项的至少 70%（如：修复了 V1 中的崩溃问题、增加了数据持久化、优化了交互流程）


**第 8 周 · 课 15：自由冲刺日（三）**

- 全程（工程师 + 助教巡场）：由助教组织"闪电解惑"环节——各组列出当前最困扰的 1 个技术问题，工程师在 30 分钟内集中解答。剩余时间继续冲刺
- 硬件/产品设计工程师注意到有团队开始打印更完整的外壳或支架，主动在助教处建立"打印进度白板"，供各团队登记打印任务，避免设备拥堵
- **本周衡量产出**：各组在课 16 前完成 V2 版本的功能冻结（不再新增功能，只修复 bug）。助教确认各组产品功能清单是否已锁定


**第 8 周 · 课 16：里程碑检查 · V2 版本演示**

- 全程（主讲老师主持 + 工程师 + 外部资本代表 + PR 部门代表到场）：每组进行 12 分钟演示，重点展示——V1 反馈的采纳情况（如：投资人建议改进的交互是否已优化？PR 部门建议的品牌元素是否已融入？）；产品的稳定性与完整性（如：连续操作 3 次不崩溃、硬件响应时间在可接受范围内）；用户体验的提升（如：操作流程是否流畅、界面是否美观）
- 外部资本和 PR 部门根据"产品是否接近可交付状态"打分，并给出跟投意向。**如果团队已经为产品制作了简单的 3D 打印外壳或固定装置**，PR 部门会特别关注"产品是否具备陈列和展示的品质感"——这不作为硬性评分项，但会显著影响对"品牌潜力"维度的感性判断
- **本周衡量产出**：各组必须演示一个比 V1 更稳定、更完整、更贴近真实产品的版本（可以仍有 bug，但核心流程必须流畅）。允许使用模拟数据或原型硬件
- **关键底线**：若 V2 相比 V1 无明显进步，该组需在课后一周内提交详细改进方案并补录演示，否则 V2 阶段记为"不通过"


### 第五阶段：V3 冲刺与最终发布会（第 9–10 周）

> **本阶段核心目标**：各组完成最终版本，在技术发布会上进行完整路演和现场演示，接受四方评审并争取投产意向。
> 
> **对应"山巅"视角的深化**：学生从具体实现中"抬头"，思考更大的问题——"这个软件到底解决了谁的什么问题？如果要在现实世界里'活下来'，还缺什么？"天使投资评审环节将评分标准从"代码写得好不好"转变为"产品是否有市场价值"，让学生从"Maker（制造者）"视角切换到"Stakeholder（利益相关者）"视角。技术再强，讲不出商业价值和用户痛点，在真实世界中就难以存活。这正是"软件企业 = 软件 + 商业模式"这一课程定位的实践体现。


**第 9 周 · 课 17：自由冲刺日（四）**

- 全程（工程师 + 助教巡场）：最后冲刺。工程师提供"发布会前特训"——各组可预约工程师进行 30 分钟一对一"压力测试"（模拟发布会现场可能遇到的技术刁难）
- 主讲老师观察各组进展，对有困难的团队进行心理疏导和策略调整。硬件/产品设计工程师开启"发布会展台快闪服务"——中午 1 小时，团队可带着 3D 模型文件来咨询，工程师协助确认切片参数、预估打印时间，确保展台陈列品能在发布会前就位。**此项服务完全按需预约，不做任何强制**
- **本周衡量产出**：各组完成 V3 版本的核心功能（即"如果只能演示 3 分钟，必须展示的那部分"），并准备好发布会演示预案（Plan A 和 Plan B）


**第 9 周 · 课 18：Beta 预演 & 发布彩排**

- 全程（主讲老师 + 工程师 + 助教 + 外校大众评审代表模拟"最挑剔观众"）：每组进行 8 分钟压缩演示（严格计时），只展示最核心的价值。模拟结束后，评委只给"攻击性反馈"——尖锐指出演示中可能翻车的地方
- 各组记录反馈，连夜调整演示策略。硬件/产品设计工程师在预演结束后提示：如有团队制作了 3D 打印展品，建议在发布会前拍照留档，用于学习闭环报告中的"前后对比"
- **本周衡量产出**：各组产出一份《发布会演示脚本》（包含：谁来说、怎么演、紧急预案），并在 GitHub 上发布最终的产品宣传视频（30 秒）


**第 10 周 · 课 19：最终准备 & 展台布置**

- 全程（全体师生 + 外校远程参与者）：各组在教室布置专属展台——摆放硬件样品、易拉宝海报、二维码（可链接到 GitHub 仓库）。工程师协助进行最后一次硬件调试
- 助教组织"观众预体验"——邀请其他组同学和外校在线参与者轮流体验产品，提出"最后时刻的意见"。鼓励有条件的团队在展台上额外陈列 3D 打印的文创周边、产品概念模型或 Logo 立体字，增强展台视觉吸引力（**这一项非必须，但实践中往往显著提升路演前的第一印象**）
- **本周衡量产出**：展台布置完成，硬件运行稳定，演示设备（投影、备用电源、手机热点）准备就绪。各组提交《发布会最终演示 PPT》


**第 10 周 · 课 20：中关村学院"技术发布会"（评审 #3 · 总决赛）**

全程（主讲老师主持 + 院领导 / 外部资本 / PR 部门 / AI 商学院代表到场）：

- 每组 15 分钟路演 + 5 分钟 Q&A（共 20 分钟 / 组）。路演必须包含现场演示（不允许只播放视频），即便硬件出问题，也要展示"应变能力"
- 四方评审团当场评分——技术实现（工程师评分）占 25%，文创 / 品牌价值（PR 部门）占 25%，商业 / 投资价值（外部资本）占 25%，AI / 创新性（AI 商学院）占 25%。评审团被鼓励在"文创/品牌价值"维度中，将产品的**"物理呈现品质"**（如展台布置、产品外观完成度、是否具备陈列感）作为感性参考，但不设硬性分值
- 现场颁发五大奖项：最佳文创产品奖（PR 部门冠名）、最佳投资价值奖（外部资本冠名）、最佳 AI 创新奖（AI 商学院冠名）、最佳工程实现奖（工程师团队冠名）、最佳失败复盘奖（主讲老师冠名）
- 现场签署"投产意向书"：PR 部门宣布采购订单，外部资本宣布跟投意向，AI 商学院宣布孵化意向
- **本周最终衡量产出**（即课程核心产出）：
  1. 每个团队必须成功演示一个可运行的软硬件产品（至少达到"功能原型"级别）
  2. 所有团队的代码、设计图、演示视频已提交至 GitHub 公开仓库
  3. 每位学生完成并发布"学习闭环报告"个人博客（正式提交截止日期设在第 14 周末，但第 13 周总结会上需有草稿或大纲）


### 第六阶段：硬件打样与深度孵化（第 11–12 周）

> 说明：这两周为硬件打样等待期，同时也是优秀项目的深度孵化期。课堂教学转为更灵活的辅导和跟进形式。
>
> 对应"山巅"视角的延续：学生继续思考创新的规律——技术成熟度曲线（Hype Cycle）适用于什么场景？如何跨越创新的鸿沟？成功地把新技术推向更广大的市场需要什么？

- **硬件打样跟进**（工程师 · 硬件/产品设计 主导）：获奖团队的硬件设计进入打样流程，工程师辅导解决 PCB 制板、外壳 3D 打印、供应链对接等实际问题
- **项目深度孵化**（主讲老师 + 工程师 · 软件/AI + 工程师 · 硬件/产品设计）：辅导有意愿继续打磨的团队，优化代码、完善文档、准备产品化。工程师可为需要精致外壳的团队提供进阶 3D 打印指导（如光固化打印、表面处理、装配设计等），帮助产品从"原型"迈向"可量产外观"
- **补强辅导**（助教组织）：针对跨专业博士生，根据"疑问热力图"安排专题小灶课，查漏补缺
- **外校学员互动**：集中回复 GitHub Issues 中积累的问题，整理成 FAQ 文档。3D 打印模板库根据学期中学生的实际使用反馈进行修订和扩充，并作为课程开源资产发布
- 关键产出：硬件打样订单发出；项目孵化计划确定；FAQ 文档更新；3D 打印模板库 v1.0 发布


### 第七阶段：学习闭环总结会 + 课程复盘（第 13 周）

> 对应"学习闭环"的完成——从学期初的提问到学期末的回答，学生完成了一次完整的认知循环。正如英国哲学家培根所说："若始于怀疑，必终于确信。"通过"开学叩问-实践求解-期末回望"的方式，学生得以在真实项目中不断验证和修正自己的疑问，而不是从书本或老师那里找标准答案。同时，这一阶段也体现了课程的"千帆竞发"精神——所有团队的代码、设计图、演示视频已提交至 GitHub 公开仓库，形成一幅完整的"学习生态图"，每一艘"船"的航迹都清晰可见、可供后来者参考。


**第 13 周 · 课 21：学习闭环总结会（上）—— 回答我的 n 个问题**

全程（主讲老师主持 + 工程师 / 助教 / 外部嘉宾到场）：

- 前 20 分钟（主讲老师）：回顾学期初的"快速阅读与提问"作业，展示当年的"疑问热力图"，说明本次总结会的意义——完成"提问 → 实践 → 反思 → 回答"的认知闭环
- 中间 50 分钟（学生展示）：随机抽取 6–8 位同学（或自荐），每人 5–6 分钟，逐一回答自己学期初提出的 5 个问题，重点分享——哪些问题在课程中找到了明确答案？具体来自哪段经历？哪些问题有了全新的理解？认知是如何转变的？哪些问题至今仍未解决？新的困惑是什么？
- 最后 20 分钟（嘉宾互动）：工程师、助教或到场的外部嘉宾对学生的回答进行点评，或补充回答一些共性问题。**3D 打印相关的体验分享**自然融入学生回答中——那些选择使用过 3D 打印的同学可以讲述：从"设计图"到"实物"的转化如何改变了他们对"产品"的理解


**第 13 周 · 课 22：学习闭环总结会（下）—— 课程复盘与展望**

全程（主讲老师主持 + 全体学生参与）：

- 前 40 分钟（分组讨论）：各团队内部进行"事后诸葛亮"会议，复盘本团队的得与失，每位成员分享个人最大的三点收获
- 中间 30 分钟（全班交流）：每组推选一名代表，分享本组的关键发现与感悟（5 分钟/组）。鼓励使用**"设计稿 vs 实物"对比照片**作为可视化素材，直观展示从"纸面"到"现实"的转化过程
- 最后 20 分钟（主讲老师总结）：
  - 串联全书核心概念与课程各阶段的对应关系
  - 展示从学期初到学期末的"成长轨迹"（如：GitHub 提交热力图、博客阅读量、问题质量的变化等）
  - 发布"学习闭环报告"的正式提交要求（课后一周内提交个人博客）

关键产出：

- 每位学生的"学习闭环报告"草稿或大纲
- 团队的"事后诸葛亮"会议记录
- 班级整体的课程复盘总结（由助教整理归档并开源）


### 第八阶段：结课缓冲（第 14 周）

**第 14 周 · 缓冲（无正式课程安排）**

- 学生提交最终的"学习闭环报告"个人博客（截止日期设在第 14 周末）
- 主讲老师 / 助教完成所有评分汇总
- 课程全部资料最终归档
- 发布 GitHub 仓库的最终版本，更新 Contributor 名单和荣誉贡献者名单
- 外校学员 FAQ 汇总发布


## 第三部分：每周答疑课（60 分钟）

> 定位：前 30 分钟为"构建之法读书会"，后 30 分钟为自由答疑 + 基础补强。跨专业博士生可在 14 周内系统读完《构建之法》，建立软件工程知识框架。每次答疑录播将上传 GitHub，供外校同学同步学习。
>
> 在答疑课中，主讲老师和助教将与学生一起回顾"千帆竞发图"，动态展示所有同学的进度。老师可以教给学生一个论点，但无法替学生完成体验和感悟——学生通过亲手劳作和近距离观察获得的这些体验与体会，才是这门课真正的收获，远比"软件工程很重要"之类的口号更能留在他们心里。


**第 1 周 · 软件工程概论 + 结对入门**
- 前 30 分钟：《构建之法》第 1 章"概论"——什么是软件工程，理论 vs 实践
- 后 30 分钟：Git / GitHub 基础操作教学（外校可同步跟学）
