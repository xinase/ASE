# 第1章　概论

理论是灰色的，而生命之树常青   —歌德

## 1.1 软件 = 程序 + 软件工程

几乎所有的程序员都知道“程序＝数据结构＋算法”这句名言。实际学习和工作中，人们常常产生疑问：数据结构和算法在真实软件开发中到底有多大用处？程序如何一步步演变为复杂的软件服务？AI时代，软件是否需要加上“数据”和“模型”？商业模式又如何影响软件需求？

（本小节包含阿超四则运算程序的故事，以及从简单程序到软件服务、商业模式的完整演进过程。）

## 1.2 为何需要软件工程

### 1.2.1 软件的固有难题

软件具有复杂性、不可见性、易变性、服从性、非连续性等本质特征。这些特征使得单纯的个人手工开发难以应对。AI工具虽然提高了局部效率，但也带来了新的挑战。

### 1.2.2 成熟度的对比——对比从纸飞机开始的航空业

航空业从纸飞机、业余爱好、探索阶段，逐步发展到成熟产业和航天高峰。软件行业与之类似，也经历了不同发展阶段。AI正在推动软件进入新的创新浪潮。

### 1.2.3 软件工程的定义

软件工程是把系统的、有序的、可量化的方法应用到软件的开发、运营和维护上的过程。它包括需求分析、设计、构建、测试、维护等核心领域，以及流程和工具的支撑。

## 1.3 软件工程的目标：在约束条件下创造“足够好”的软件

### 1.3.1 Bug，Feature和“足够好”

Bug是软件行为与设计者或用户期望不一致的结果。好的软件不是追求零Bug，而是需要在时间、成本等多种约束下做出权衡，达到“足够好”的状态。

### 1.3.2 如何衡量足够好？

通过Validation（确认软件满足用户真实需求）和Verification（用正确的方法开发）来衡量。还需考虑功能安全性，确保意外情况下有兜底机制。

### 1.3.3 AI时代的“足够好”

AI生成代码带来了速度提升，但也引入了确定性、透明度和质量幻觉等问题。人类仍需负责意图对齐、可维护性、安全和合规性。

## 1.4 软件工程的广阔天地：知识领域与内涵

软件工程知识体系可提炼为三大支柱：核心生命周期（构建正确软件）、支撑性基础（确保好用与耐用）、过程与约束（高效协作）。超算项目是典型案例。

## 1.5 软件工程与计算机科学

计算机科学侧重发现长期客观真理，软件工程侧重短期的实际结果和折衷。两者相互支撑，但侧重点不同。

## 1.6 总结：软件工程师的宗旨

软件工程师的宗旨是“我构建，故我在”。AI时代，工程师从“码农”逐步升级为“意图的最后守门人”。项目可分为Build To Learn、Build To Show、Build To Serve、Build To Win四类。
