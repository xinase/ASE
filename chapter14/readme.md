## 14.1 什么是软件的质量

软件质量不再是“浪漫的完美交付”，而是“在指定条件下满足显性与隐性需求的能力”。它由三部分构成：**程序质量**（外在功能体现）、**软件工程质量**（过程与体系的可见性、风险控制与可验证性）以及**数据质量**（AI时代新增）。高质量软件需同时兼顾“好、快、便宜”，并通过持续验证（Verification）与确认（Validation）来保障。

## 14.1.1 程序的质量：外在功能的体现

程序质量体现在功能正确性（Precision/Recall）、性能（速度、吞吐量、并发）、用户体验、安全性与国际化等方面。可通过NPS、MTTF/MTBF等量化指标衡量。

## 14.1.2 软件工程的质量：过程与体系要实地演练

软件工程质量聚焦过程的可见性、风险控制、内部产物质量、成本控制与可验证指标。GitLab数据误删事故揭示：再多的备份机制，若未实地演练（灾难恢复测试），也只是“写在纸上”的质量。

## 14.1 “面向用户进行测试”的能力对于团队中的测试人员来说是否是必须的？

学生发现招聘网站对测试工程师的要求多集中于测试方法，很少提“面向用户测试”，询问这种能力是否必须，懂得用户体验的测试人员是否冗余。

**问题的核心：** 测试人员不能只做Verification（验证规格），还要做Validation（确认用户真正需要什么）。这是测试与QA的本质区别。

**回答：**

书里14.1节区分了“程序质量”和“软件工程质量”，而14.2节明确指出：测试只是QA的一部分，QA还要从用户角度确认产品是否解决了真实问题。

一个只对着Spec验证的测试员，发现不了“功能全对但用户恨得要死”的问题。这不是冗余，而是失职。招聘网站上没写，不代表不需要——因为大多数公司连QA和Test都没分清。真正成熟的团队，测试人员必须懂用户体验，否则谁来发现“纸割伤”（12.3.3节）？

**扩展思考：**

书里14.2.2节提到“质量不能全部外包给陌生人”。同样，用户体验也不能外包给产品经理一个人。测试人员从用户视角出发，能发现PM和开发都忽略的细节。特斯拉让设计师管质量，正是这个道理——质量是设计出来的，不是测出来的。

## 14.2 软件的质量与“好”、“快”、“便宜”相关，如何平衡三者？

学生询问如何在开发中平衡“好、快、便宜”，以及采用CMMI是否能达到最佳值。

**问题的核心：** 三者不可兼得，只能根据场景取舍。CMMI不是银弹。

**回答：**

书里9.1节已经回答了这个问题：大部分优秀团队只能做到三个中的两个。

- **好 + 快**：不便宜（堆人力、用成熟框架）
- **好 + 便宜**：不快（开源、慢慢磨）
- **快 + 便宜**：不好（MVP、技术债）

CMMI不能直接给你“最佳值”，它只是帮你建立可重复、可预测的过程。CMMI三级以上的团队，能稳定地交付“好”的产品，但“快”和“便宜”取决于业务选择。

**平衡点的决定因素：**
- 市场窗口期短 → 牺牲“好”，先快
- 安全攸关系统 → 牺牲“快”和“便宜”，先好
- 内部工具 → 便宜优先

**扩展思考：**

书里14.1.5节“质量的成本”给出了经济学视角：左移预防成本低，右移故障成本高。但在AI时代，现代左移（经验加速）让“快”和“好”可以同时提升——因为AI大幅降低了构建和测试的成本。三者的矛盾在缓解，但不会消失。

## 14.3 CMMI为什么在以互联网业务为主的中小企业效果还有待观察？

学生询问CMMI在互联网中小企业中效果不明显的原因及当前普及情况。

**问题的核心：** CMMI是重量级流程体系，与互联网中小企业的“快速试错、灵活应变”文化存在冲突。

**回答：**

书里14.1.3节明确说：CMMI在传统软件企业成效不错，但在互联网中小企业效果待观察。原因：
1. **太重**：CMMI要求大量文档、评审、追溯，中小企业没这个人力。
2. **太慢**：互联网业务要求小时级迭代，CMMI的变更管理周期太长。
3. **文化冲突**：CMMI追求“可预测”，互联网追求“拥抱变化”。

**普及情况**：目前CMMI多用于大型外包、汽车、航天等需要竞标或合规的领域。互联网中小企业更常用DevOps、敏捷、混沌工程等轻量方法。老师回复里也说了：很多成功公司实质上用了类似CMMI的思想（如持续集成、门禁、复盘），但不会去拿证。

**扩展思考：**

书里14.1.3节对比了SpaceX的“关注工具而非规则”。对中小企业，投资CI/CD工具、自动化测试、代码审查门禁，比通过CMMI认证更务实。CMMI五级追求“量化管理”，但互联网公司用A/B测试和实时监控，本质上也是量化——只是手段不同。

## 14.4 区块链等去中心化技术是否可以为所有产品提供独立质量认证？

学生认为“人肉认证”（创始人自己用）不可靠，询问区块链等去中心化技术能否提供独立质量认证。

**问题的核心：** 技术不能解决信任的根本问题——信任的核心是“验证能力”和“问责机制”，不是记录不可篡改。

**回答：**

书里14.2.1节列举了网站使用明文密码、OpenSSL心脏出血等案例，说明“人肉认证”和“开源=安全”都是幻觉。区块链能保证记录不可篡改，但无法保证记录的内容本身是正确的。一个未经验证的测试报告，上了链还是假的。

独立质量认证需要三样东西：
1. **权威标准**（如ISO 26262、ASPICE）
2. **专业机构**（有资质做审核）
3. **法律问责**（出事了有人负责）

区块链能辅助第1项（存证），但无法替代2和3。正如老师回复所说：如果你连政府和机构的认证都怀疑，凭什么相信一个匿名节点？

**扩展思考：**

真正有价值的不是“去中心化”，而是“可验证性”。开源代码的可验证性来自任何人都能审查、编译、测试。区块链的“智能合约审计”也是由专业安全公司做，不是靠链本身。所以，去中心化技术可以作为认证流程的一部分（如存证、溯源），但无法取代独立、专业、有问责的第三方认证体系。
