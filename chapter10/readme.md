# 第10章　典型用户和场景

要完成的任务本身就包含了场景，而场景（context）才是关键。  —克莱顿·克里斯坦森，创新专家

## 10.1 典型用户和典型场景

典型用户（Persona）是基于真实用户调研构建的虚拟人物，用于代表核心用户群体的特征、动机和痛点。结合Job To Be Done（JTBD）理论，能帮助团队从“用户需要什么功能”转向“用户想完成什么任务”。在AI时代，Persona的定义、验证和迭代都可被AI大幅加速，同时必须新增“AI安全威胁用户”（如数据投毒者、提示词注入者）等新型典型用户。

### 10.1. Persona 何时“下线”？是否需要可操作的撤换门槛？

讨论如何设定客观、可量化的标准（如任务失败率、可用性测试数据）来决定是否要修改或放弃一个已有的典型用户（Persona），避免Persona成为摆设，也避免凭感觉决策。

**问题核心**：在课程项目等有限资源下，如何设定一个可操作、低成本的判定阈值，来触发Persona的撤换与场景重构？

**回答**：  
书中10.1.3节“被删除的典型用户”明确指出：团队因实地采访发现“教导主任老吴”并不使用系统，果断将其从典型用户中删除。这个案例的核心精神是**敏捷开发与用户研究的核心价值：快速证伪**。证明一个假设是错的，比证明它是对的更有价值。

在AI时代，可以引入定量和定性指标形成可操作门槛：
- **定量指标**：针对该Persona的核心JTBD，定义“任务完成率”和“任务完成时间”。例如，连续5个代表“莫尔特”的用户在AI辅助下无法在10分钟内完成核心任务，即触发评审。
- **定性指标**：用户访谈、支持工单或应用内反馈的集中负面评价。
- **最低证据门槛（课程适用）**：您提出的“5人可用性测试 + 关键日志事件”是非常务实的标准。5位被试者完成核心任务，记录失败次数和异常日志，即可作为撤换的客观依据。

**扩展思考**：  
未来AI可以持续分析用户行为数据，自动识别出与现有Persona都不匹配的新用户聚类，并主动建议团队创建新的Persona，实现Persona的动态演化。

### 10.2. “用户没这么笨” vs “用户需要很多帮助”：如何界定？

开发团队内部对用户能力存在矛盾假设（高估或低估），如何通过Persona和场景分析来达成共识，做出合理的设计决策？

**问题核心**：团队争议的本质是每个人心中默认的“用户”不同。如何界定、如何统一？

**回答**：  
书中10.1节通过Visual Studio的三个Persona（Mort、Elvis、Einstein）给出了答案：**不是二选一，而是细分**。
- “用户没这么笨” → 对应“艾尔维斯”和“爱因斯坦”，需要高度定制和掌控力。
- “用户需要很多帮助” → 对应“莫尔特”，需要傻瓜式引导和大量示例。

界定方法：
1. **建立Persona**：将抽象用户具象为有名字、目标、痛点的人。
2. **针对设计**：讨论功能时问“这个功能对莫尔特有帮助吗？爱因斯坦会怎么用？”。
3. **提供不同交互路径**：同一系统可同时提供图形界面和命令行，满足不同Persona。

**扩展思考**：  
AI时代可设计**自适应交互界面**。系统通过分析用户行为（如快捷键使用频率、求助次数）动态推断用户类型，自动调整提示详细程度和功能默认值，实现“一人一面”。

### 10.3. 专业性产品是否需要考虑非专业用户？

像Visual Studio这样的专业工具，是否应该为逐渐增多的初学者（非专业用户）牺牲一部分专业体验？

**问题核心**：专业产品的核心用户与潜在大众用户之间的需求矛盾，如何平衡或化解？

**回答**：  
书中10.1节指出：“我们的软件不是为所有人服务的。”但这不等于“绝不考虑非专业用户”。关键在于**定义核心Persona及其优先级**。

对于Visual Studio，核心是“艾尔维斯”和“爱因斯坦”。但微软通过创建**VS Code**解决了矛盾：VS Code轻量、插件化，既服务“莫尔特”，也通过插件服务专业开发者。这是**市场细分与产品线策略**——用一个新产口覆盖被原产品“排除”的用户，而非强行塞入同一个产品。

因此：
- 短期/单一产品：必须做出取舍，优先服务核心Persona。
- 长期/公司战略：当非专业用户群体足够大时，应创建**定位不同的新产品**。

**扩展思考**：  
AI代码助手（如GitHub Copilot）正在模糊界限。非专业用户可借助AI生成复杂脚本，专家则用AI自动化重复劳动。AI使得同一工具界面能以不同“智能层次”服务不同用户。

### 10.4. 为什么要分析不受欢迎的用户（如捣蛋鬼）？

质疑单独为捣乱者、作弊者建立Persona的必要性，认为其需求可归入安全需求，单独分析投入产出比不高。

**问题核心**：不受欢迎的用户分析是否值得？能否被安全需求覆盖？

**回答**：  
书中10.1.2节“捣蛋鬼阿狗”的案例明确回答：**非常有必要，这是从被动响应到主动设计的思维转变**。
1. **前置安全设计**：没有“阿狗”，安全是事后补丁，易与用户体验冲突（如给所有人强验证码）。有“阿狗”，团队在设计阶段就思考“这个API阿狗会怎么攻击？”，实现内建安全。
2. **统一团队心智**：“阿狗”成为具体假想敌，功能优先级判断标准统一：“这能防住阿狗吗？”。
3. **优化正常用户体验**：正因为有针对性防御，才能对99%正常用户简化流程（如仅对可疑操作弹验证码）。

您提到的“微信起初没把淘宝链接视为不欢迎”，恰恰说明**不主动定义不受欢迎用户的风险**——问题会晚到但代价更高。

**扩展思考**：  
AI时代，“阿狗”进化为“数据投毒者”、“模型窃取者”、“提示词注入者”。为其建立Persona，可引导设计AI防火墙、差分隐私、输入输出过滤等，实现从功能安全到AI安全的演进。

### 10.5. 防破坏的“度”在哪里？该为多大努力的黑客调整设计？

以一个在线考试系统为例，讲述过度防御导致普通用户体验极差，询问如何在防御恶意用户和保证体验之间权衡。

**问题核心**：应该以多大程度的攻击努力作为防御设计的边界？如何避免“自己吓自己”？

**回答**：  
这个案例精准展示了**瀑布式开发的弊端**——试图在考试前一次性构建完美防御。正确的“度”应通过**敏捷迭代**来校准：
1. **发布MVP**：第一版只做最简单的静态试卷 + 拍照上传。核心JTBD是“成功提交答案”。
2. **识别真实威胁**：通过小范围测试发现实际作弊行为（可能根本不是枪手，而是互相传答案）。
3. **增量添加防御**：每次只加最必要的一层（如随机打乱题序），然后观察对正常用户的影响。
4. **设定“可接受”的代价**：没有100%安全。防御成本（开发+体验损害）不应超过被阻止攻击的潜在损失。无法量化时，就用最小成本快速验证。

**“度”的答案**：防御措施的成本（包括用户体验成本）不应超过被阻止攻击的损失。用真实反馈而非想象来驱动。

**扩展思考**：  
AI可扮演“红队”，自动尝试各种攻击（SQL注入、提示词注入、DDoS模拟），并评估攻击成功率和成本。同时AI也可模拟普通用户，测试防御机制对体验的损害，为权衡提供数据支持。

### 10.6. 当“用户”就是你自己，软件工程还有必要吗？

对于个人开发、自用脚本，需求文档、测试用例、版本规划等工程实践是否冗余？价值何在？

**问题核心**：自用型开发中，工程纪律的必要性和简化形式。

**回答**：  
即便用户只有自己，软件工程的**思想和部分实践**依然必要，但形式可简化。价值在于：
1. **未来的可维护性**：6个月后你可能完全忘记实现细节。没有版本控制和简单测试，一个小改动可能引入大bug。
2. **训练工程习惯**：写单元测试、版本控制会内化为职业素养，未来协作时无缝切换。
3. **应对意外成功**：小工具可能突然被他人使用，基本工程健壮性会放大其价值。

简化实践：
- **需求**：用一两句话写JTBD。
- **设计**：画个简单流程图。
- **测试**：手动覆盖核心路径和若干边界条件。
- **版本控制**：**绝对必要**。`git init` + 定期commit是成本最低、回报最高的实践。

**扩展思考**：  
这就是“Docs-as-Code”和“个人知识管理”的实践。当使用AI编程助手时，你与AI的交互过程本身就是一种需求分析与验收测试。

### 10.7. 服务于小部分典型用户的idea值得鼓励吗？

在课程项目或创业中，只服务小众但定义清晰的用户群体的想法，是否值得投入？当不被外界看好时怎么办？

**问题核心**：聚焦小众用户与商业价值/课程评价之间的矛盾。

**回答**：  
书中10.1.3节明确支持：**“我们宁可从小部分人出发，要非常明确地定义出谁是我们的用户。”** 答案是**绝对值得鼓励，尤其在软件工程课程中**。
1. **课程目标**：课程的核心是经历完整开发流程。服务小众垂直领域比服务百万人的简单应用更能实践需求分析、领域建模和迭代。
2. **商业价值**：这正是“利基市场”策略。对于学生项目，学习价值和创新性优先于商业价值。
3. **面对外部阻力**：这是锻炼“讲故事”和“捍卫设计决策”的机会。你需要清晰解释：这个小众用户是谁、有何痛点、为什么现有方案不好、为何服务好他们能成为更大市场的起点。

**扩展思考**：  
AI时代，服务“小部分人”的成本急剧下降。通过微调或提示词工程可快速创建服务超细分领域（如“法律合同审查智能体”）的产品。**AI让利基市场变得有利可图**。

### 10.8. “真的有用户嘛？”——课程项目的用户困境

课程项目结束后不再维护，那么项目期间的“真实用户”和“用户量”是否有意义？如何获得真正的、可持续的用户？

**问题核心**：课程项目用户价值的本质，以及获取真实用户的可操作方法。

**回答**：  
意义是有的，但需要**重新定义“用户”**。在课程语境下，用户的价值不在于长期留存，而在于**提供真实反馈以驱动开发**。
- **短期真实用户**：同班同学、其他专业同学、找来的5个测试者，就是真实用户。让他们在2-4周内真正使用产品核心功能，其困惑和吐槽就是最宝贵的财富。
- **用户量的意义**：不在于商业价值，而在于验证系统负载能力和产品普遍性。

更易实施的方法：寻找一个**有痛点且愿意花30分钟反馈的微型组织**，例如：
- 为学校社团做活动报名小程序。
- 为附近咖啡店做积分兑换页面。
- 为课题组做文献共享插件。

这些用户需求真实、紧迫，愿意投入时间。

**扩展思考**：  
**“产品死后验尸报告”** 本身就是有价值的工程实践。项目下线后，分析用户画像吻合度、流失原因，这种复盘能力是大厂面试官看重的。

### 10.9. 规格说明与代码“同源”：最低实践是什么？

在资源受限的课堂项目中，如何低成本地实现“Docs-as-Code”（文档即代码），保持Spec与代码同步？

**问题核心**：可行的最低实践清单，区分必做项和可选项。

**回答**：  
书中10.3.1节强调“活文档”和“文档腐烂”问题。在课堂项目中，可设定**最低可行清单**：

**必做项（核心）**：
1. **同库管理**：用Markdown写`README.md`和`docs/`，与代码同Git仓库。
2. **关键场景的Given-When-Then**：对每个核心用户故事，用`Given-When-Then`格式写出2-3个验收标准。
3. **PR关联**：提交PR时，必须链接到Spec文档章节或`issue`编号。审查者检查代码是否符合Spec。
4. **一个自动化测试**：至少为核心场景编写一个端到端自动化测试（如Playwright）。当测试通过时，证明代码“基本上”符合文档。

**可选项（锦上添花）**：
- CI链接检查（检查Markdown中的所有链接是否有效）。
- 基础结构检查（脚本验证README中的API端点在代码中是否存在）。

**扩展思考**：  
AI可极大降低Docs-as-Code门槛：让AI根据代码变更自动更新Spec初稿；或在CI中加入AI评审步骤，检查代码与Spec的不一致。

### 10.10. 需求分析是否最容易被忽视却最致命？

为什么学生项目往往在写代码后才发现需求理解错误，导致大量重构？如何改善？

**问题核心**：如何在不虚构用户、时间紧张的环境下，做出高质量的需求分析？

**回答**：  
IBM统计56%项目失败与需求不清有关，书中也说“场景才是关键”。**需求分析不是“最容易被忽视”，而是“最容易做不好”**。多数人错误地认为“我想清楚了”等于“需求清楚了”。

在学生项目中，核心方法不是“学会观察人”，而是**学会验证假设**：
1. **接受错误**：接受需求文档第一版99%是错的，这是开始。
2. **纸面原型验证**：写代码前用纸或PPT画界面，找同学模拟操作。这能消灭50%的逻辑冲突。
3. **写用户故事，不写功能列表**：故事强制思考“谁，要什么，为什么”。
4. **用TDD反向驱动需求**：写代码前先写失败测试用例，测试的输入/输出/边界条件就是需求分析的过程。

**扩展思考**：  
**AI是最佳需求分析搭档**。动手前可以问AI：“我要做图书馆管理系统，典型用户是大二学生小明，他的JTBD是快速找书。请列出5个悲伤场景，并为每个场景生成Given-When-Then验收标准。” AI不会给完美答案，但会提供一个充满细节的框架，让你快速进入验证阶段。

### 10.11. Spec是否应说明已知缺陷和副作用？

规格说明书（Spec）中是否应该写明软件的已知缺陷和功能的副作用？如何区分？

**问题核心**：副作用与缺陷的本质区别，以及各自在文档中的位置。

**回答**：  
必须**明确区分“副作用”和“缺陷”**。

- **副作用**：为实现核心目标而产生的计划内、非预期但可接受的后果。**必须在Spec中明确说明**。例如：“AI批改简答题效率提升极大，副作用是对于开放性问题，评分可能与老师有5%偏差，需人工复核。” 这是对功能边界和风险的诚实声明。

- **已知缺陷**：与Spec预期行为不符的计划外错误。**不应在Spec中作为功能描述**，而应在Release Notes或项目看板的“Known Issues”区域列出。根据书中15.1章，团队可根据严重程度决定带着已知但不致命的缺陷发布，以换取快速反馈。

**扩展思考**：  
AI系统的“概率性”使“缺陷”和“副作用”界限模糊。AI产品的Spec应包含**“AI行为与性能说明书”**，明确描述预期准确率、不确定性展示方式、用户反馈机制等。将AI的不确定性作为特性而非缺陷来定义，是AI时代Spec的核心挑战。
