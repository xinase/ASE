## 2.1 单元测试

单元测试是软件工程师的基本技能。即使在AI工具高度辅助的今天，工程师仍需对模块质量负责。单元测试能显著降低后期修复Bug的成本，是保障模块清晰、可维护的核心手段。

### 2.1.1 单元测试的讨论

通过对话形式说明为什么要在设计阶段就考虑单元测试，以及单元测试如何帮助明确API语义、记录设计变更。

### 2.1.2 单元测试和TDD

介绍TDD的“红—绿—重构”循环。AI工具可辅助生成测试用例和代码，但开发者仍需对测试负责。给出AI辅助TDD在不同场景下的适用性分析。

### 2.1.3 单元测试的标准

好的单元测试应具备准确性、独立性、可重复性，并覆盖所有代码路径（包括错误处理）。单元测试代码必须与产品代码同步维护。

### 2.1.4 回归测试

在单元测试基础上建立回归测试，确保新代码不会破坏已有功能。回归测试应自动化，以便在每次构建时快速验证。

### 2.1.5 测试覆盖率——函数/行/分支/修改条件

详细说明不同覆盖率层次（函数、语句、分支、MC/DC）的含义和实际价值。通过具体案例说明100%分支覆盖仍可能存在Bug，以及将复杂条件拆分为简单if-else的好处与代价。

# 第2章 个人技术和流程：核心问答汇编

> 解答在 AI 辅助编程时代如何进行质量控制、效能优化与流程管理。部分提问来源： https://bbs.csdn.net/forums/SoftwareEngineering?typeId=828  

## 2.1 在设计阶段就写单元测试，会不会导致过度设计？探索性项目连全貌都不清楚怎么测？

**回答：**
在设计阶段写单元测试，不是为了写死实现细节，而是为了强制定义“API 契约”。最好是在设计的时候就写好单元测试，这样单元测试就能体现 API 的语义 。

对于原型验证、UI 视觉设计或探索性编程，确实不适合严格的测试驱动开发（TDD）。在 Demo 阶段，你可以先跑通主干；但当框架雏形确立，要封装成一个通用“单元”供业务调用时，必须补齐测试来固化边界。这绝非过度设计，而是系统具备可扩展性和防劣化的底线。

## 2.2 AI 也会照抄诸如“Zune 变砖”的经典 Bug。在 AI 时代，如何利用 AI 生成测试并规避“质量幻觉”？100% 覆盖率还有意义吗？

**回答：**
100% 的代码覆盖率并不等同于 100% 的正确性 。覆盖率工具对于“应该写但是没有写的代码”（如未释放内存、未处理的网络异常等）无能为力 。Zune 播放器在 2008 年底大面积死机，正是因为代码在闰年最后一天（`days == 366`）的边界情况陷入死循环 。AI 极易在生成的样板代码中带入此类逻辑陷阱。

**破局方法：**

1. **明确责任：** 可以让 AI 工具来写单元测试，但是，程序的作者还是要对单元测试负责 。


2. **强制质量门禁：** 在源码控制系统（如 GitHub）中配置自动化流水线，只有通过编译、单测并达到覆盖率阈值的 PR 才允许合入 。

3. **收紧边界条件：** 工程师必须显式向 AI 提出边界校验要求，确保测试用例覆盖完整的等价类边界：{边界条件 – 1，边界条件，边界条件 + 1} 。由于输入 0 对应 1980/1/1，输入 365 对应 1980/12/31（仍在 1980 年），输入 366 对应 1981/1/1，这类精确的索引临界点往往就是 Bug 爆发的高危区。



## 2.3 为了达到 MC/DC（修改条件/判定覆盖），把简洁的布尔表达式拆成冗长的嵌套 if-else，这是不是牺牲“简洁性”的无奈妥协？

回答：这不是妥协，而是工业级安全的要求。MC/DC 要求证明复合条件中的每一个子条件都能独立地影响最终的判定结果 。单纯的 100% 分支覆盖率很容易给人虚假的安全感，隐藏短路逻辑引发的致命错误 。

拆分复杂逻辑确实会大大降低程序的可读性 。因此，最佳实践是 **Keep It Simple**，尽量写简单的条件判断 。如果你发现一个布尔表达式复杂到难以测试，工程上的正确做法是重构业务逻辑或数据流，而不是在复杂逻辑上强行盖违章建筑。

## 2.4 单元测试能用随机数吗？前端 UI、机器学习以及极难到达的错误处理路径怎么测？

**回答：**

1. **随机数：** 单元测试应该产生可重复、一致的结果 。如果随机数导致程序出错但无法复现，就失去了单测的意义。随机数更适合放在端到端的模糊测试（Fuzzing）中。


2. **错误处理路径：** 单元测试应覆盖所测单元的尽可能多的代码路径，包括错误处理路径 。如果某个错误处理路径极难到达，你应该考虑是否可以直接把这个错误处理拿掉 。


3. **UI 与大型机器学习：** UI 层及模型训练层无法给出严格的预期断言（Expected Result），通常不适合细粒度的单元测试 。可以将重点转移到业务逻辑剥离、组件截图对比（UI），或对数据预处理管道（Data Pipeline）进行验证。



## 2.5 遵守开放-封闭原则时，发现初期接口设计错误怎么办？另外，职业程序员写代码的时间比例为何远低于学生？

**回答：**

* **接口错误：** 面向对象设计中的接口契约并非不能更改。如果在扩展时发现底层抽象错误，必须回到 TDD 的第三步——重构（Refactor） 。


* **时间比例之谜：** 修复故障的成本，一直是随着软件生命周期的进展而呈指数上升 。职业程序员将大量精力前置到了构建需求心智模型、编写测试契约 ，并在后期进行严格的回归测试 。


* **破局建议：** 要想获得工程级成长，不要把时间耗在重复造轮子上。尝试从数据、需求、用户和构建等维度进行扩展 ，主动为项目引入复杂性，才能跨越从学生到职业程序员的鸿沟。



## 2.6 “Vibe Coding”时代，AI 生成的底层逻辑效率可能极低（如多重隐蔽循环），如何及时发现并修复这些性能盲区？

**回答：**纯靠“读代码”或“让 AI 帮我找问题”在性能调优上是完全行不通的。AI 追求最快给出可运行结果，极易生成 $O(N^2)$ 的低效逻辑。

1. **必须使用效能探查器（Profiler）：** 构造足够大的测试数据，通过 Visual Studio 等工具的 CPU 使用率报告，找出真正耗时的“热路径（Hot Path）” 。


2. **精准指挥 AI 重构：** 找到热路径中的具体函数后，向 AI 下达带有约束条件的指令（例如：“将该函数内层的 `for` 循环查找替换为 `unordered_map` 的 $O(1)$ 查找”）。
3. **用真实数据说话：** Copilot 或其他 AI 工具提供的优化建议有时在实际运行中并没有明显的改进 。必须以真实时钟时间（Clock Time）的对比结果来决定是否采纳优化。



## 2.7 深度学习和算法训练脚本，还有“单元测试”的概念吗？如何做早期的质量控制？

**回答：**
深度学习代码同样需要严密的单元测试，否则就是在无意义地燃烧算力。针对 AI 训练脚本的测试，重点在于卡住数据流的管道：

1. **张量（Tensor）维度与类型断言：** 深度学习中最隐蔽的 Bug 就是维度错位被框架“静默广播（Broadcasting）”。在核心模块输入输出端，强断言张量的 `.shape` 和 `.dtype`。
2. **单一 Batch 过拟合测试（Sanity Check）：** 抓取 2 个样本，关闭正则化与 Dropout，测试模型能否在短迭代内将 Loss 降到 0。如果降不下去，说明前向或反向传播链条存在底层的代码 Bug。
3. **梯度流向断言：** 断言核心权重层的 `grad` 属性不是 `None` 且不为 NaN，防止计算图断裂。
4. **随机种子一致性校验：** 全局固定随机种子，确保连续两次运行同一段前向传播代码产出的浮点矩阵完全一致，以此排查未初始化的变量或未被管控的随机源。
5. **数据管道 TDD：** 预处理代码（清洗、裁切、分词）没有任何随机性，必须采用传统的单元测试进行 100% 分支覆盖。


为了更全面地反映软件工程的实践，我们针对大家在衡量开发成果时的困惑，增加了关于软件度量的探讨。站在 2026 年的视角，代码量（LOC）和单纯的“完成感”已经不再是衡量工程水平的唯一尺度。

## 2.8 软件开发的工作量和质量怎么衡量呢？PSP 中衡量软件开发的工作量和质量的指标是否合适？

**回答：**
在 AI 大规模辅助编程的今天，传统的衡量标准正在发生根本性的重构。书中提到的衡量软件开发的四个基本因素：**规模（项目多大）、时间（花了多久）、质量（Bug 多少）和交付（是否按时）** ，依然是核心框架，但在 2026 年，它们的内涵已经变了：

1. **代码量（LOC）不再等于工作量：** 过去我们认为代码写得多就是工作量大。但在 AI 时代，AI 可以在几秒内生成上千行代码。如果一个工程师只会让 AI 疯狂“吐代码”而没有清晰的**心智模型（Mental Model）** ，产出的往往是“代码膨胀”后的技术债。现在的衡量标准正从“代码行数”转向“逻辑复杂度”**和**“有效功能点”。


2. **“思路有了”不等于“工作量完成”：** 正如学生反馈中提到的，很多初学者会陷入“眼高手低”的陷阱，认为自己想清楚了就等于做完了 。在工程中，只有符合规范、通过测试且能合入（Check-in）的代码才是真实的工作量。AI 可以帮你写代码，但它不能替你负责“正确性”。


3. **质量由“自动化门禁”判定，而非主观感觉：** 关于质量谁来评定，2026 年的共识是：**数据和自动化工具是比人更客观的“专家”**。
* 
**内部：** 通过**质量门禁（Quality Gates）** ，自动计算行覆盖、分支覆盖、甚至 MC/DC 覆盖。如果代码没达到阈值，PR（合入请求）直接被拒 。


* **外部：** 利用 **Telemetry（遥测数据）** 观察用户的真实使用情况。与其听用户的“主观好坏评语”，不如看“崩溃率（Crash Rate）”、“响应耗时（Latency）”和“任务完成率”。


4. **PSP（个人软件开发流程）的价值在于“预测力”：** 衡量一个工程师是否成熟，不仅看他现在写得快不快，看他能不能准确预计（Estimate）自己需要花多少时间，并最终按时交付。

**建议：** 不要迷信代码量。在 AI 时代，最好的程序员是那些能用**最精简的代码**解决**最复杂需求**，并能通过**严密的自动化测试**证明其质量的人。
