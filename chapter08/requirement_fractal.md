## 讨论正文

### 1. 软件需求中的重复，是软件最容易解决的问题

还是在超算公司的办公室。

阿超把咖啡杯放到桌上，拉过白板，在上面画了几个代表“分形”的嵌套三角形。

果冻盯着白板，揉了揉下巴，“超哥，刚才大家讨论软件复杂度的前几个帖子，特别是你提到的那个 *软件在实现时存在很多自相似的分形维度*，我听着总觉得悬在半空。为什么我在上学的时候完全没碰到过这种事？我的《软件工程》大作业还拿了九十五分呢。”

小飞斜眼看他，“你拿九十五分那个作业，题目是什么来着？”

“就是超哥现在提的这个需求啊！”果冻一拍大腿，“给小学生出五十道随机四则运算题目并批改。我当时的解法非常直接，在网页里硬编码了五十个固定的控件，一组组连着排下去。期末演示时，程序运行得极其稳定，老师当场给了高分。”

小飞翻了翻白眼，“那如果用户现在要求变了，不仅要五十道，还要能出六十道呢？”

果冻理直气壮地摊开手，“那还不简单？我把原来的项目复制一份，改名叫 `math-project-60`，在页面里再硬写十个控件凑满六十个，重新发布上线。”

“那如果用户要八十道呢？”小飞追问。

“那就再开一个项目，硬编码八十个控件。”果冻回答得不假思索，“只要肯花时间复制粘贴，这能出什么错？反正我觉得当年的老师挺好的，让我们得了高分。”

小飞深吸一部分气，眼睛翻到天花板上去了，“兄弟，你那是把服务器当成流水线车间了。如果用户明天想要单数，后天想要一千道，你打算在后台存一千个独立的项目目录？”

阿超摘下眼镜，用镜布擦了擦，转头看向果冻，“果冻，学校老师给你打九十五分，是因为当年的大作业是一张静态的试卷，老师的考核指标很单纯，交上去就封版，他并没有去考察你的程序在未来如何处理更多的需求变形。拿到满分的经验，让你误以为这就是软件开发的全部了。老师没有水平，也没有愿力去让学生学好，学生也满意地获得了高分，这也说明了，正在学习中的学生并不是评价一门课质量的唯一指标。”

阿超把眼镜戴回去，用记号笔在白板上重重地敲了两下。

“老师其实也不懂真实的软件开发。他教给你们的，最多是网页开发的 Hello World，结果你们这帮大学生浅尝辄止，拿了高分，也就这么毕业了。软件开发的最大核心，是如何应对需求的多样性和变化性。如果需求是单纯不变的、定死在五十个控件上的，那它顶多只是一个程序的函数，或者一个独立的逻辑单元，这根本不叫软件开发！”

小飞在旁边点头，手指在桌面上无意识地敲击，“确实，大作业就像是在无菌室里做标本，把逻辑卡死在某个瞬间，它当然不坏。”

“对。”阿超双手撑在桌沿，身体前倾，“一旦放到真实环境里，面对活的用户，风一吹需求就变了。果冻，你用复制项目、硬编码六十个控件的方式去对付，是在用肉身去对抗变化的概率，这也就是为什么很多人一走出校园，代码一进市场就彻底崩盘。”

阿超在白板上画了一条直线，在线上点了几个点。

“我们碰到的用户需求，有很多是基本需求的重复。软件最擅长高效地处理这类重复性的任务。如果用户的需求是在不同城市跑 $N$ 个 1000 米，我们不需要真的派人去 $N$ 个城市建操场，直接在跑步机上完成就行了。我们的‘跑步机模块’动态调用 $N$ 次就好了。”

果冻点了点头，“这个我理解，就是用循环和参数把线性的重复给消灭掉。”

### 2. 软件需求中的自相似分形

“对，但这只是第一层。”阿超把手里的记号笔换成红色的，在白板上画了一个正方形，接着在正方形里面画了一个小圆圈，小圆圈里又套了一个更小的正方形。

“更麻烦的是另外一些需求，它们并不总是线性的，而是具有自相似性的‘需求套娃’。比如我们天天用的办公软件 Word。最初的需求很简单，就是一段文字，文字有格式。接着用户说，我想在文字里插一个表格。好，表格塞进去了。然后用户又说，表格的每个单元格里还要写文字，文字还要有各种格式。再接着，用户要求在单元格的文字里，再嵌套一个表格……”

小飞听着，眼皮跳了一下，“这就变成了：文字包含格式，格式包含表格，表格包含单元格，单元格又包含文字，文字又包含格式，格式又包含表格……无穷匮也。”

阿超转过头看着果冻，“你看，这不再做是跑 $N$ 个 1000 米的问题了。这是一种分形。面对这种自相似的无限嵌套，你打算怎么写你的硬编码控件？”

小飞揉了揉太阳穴，接过了阿超的话茬：“超哥，你说的这种‘自相似的无限嵌套’，让我想起了上世纪 1980 到 2000 年的桌面办公软件时代。那时候各大巨头为了解决‘Excel 里套 Word，Word 里套 Excel’的复合文档难题，可谓是各显神通，甚至引发了操作系统级别的架构大战。”

![Word 插入表格有好几种方案](embed1.png)

![可以在 Word 文档中插入一个 Excel 的表格](image.png)

果冻顿时来了兴趣，“那时候大家都是怎么破这个局的？”

“当时不同的阵营，走出了完全不同的演进路线，”小飞一边回忆，一边在白板的另一侧梳理出几条清晰的脉络：

* **Microsoft 的路线：OLE 与 COM 组件技术**
    微软的解法最符合‘套娃’的直觉。他们推出了 OLE（对象链接与嵌入）技术。当你在 Word 里双击一个 Excel 表格时，Word 的菜单栏会瞬间变成 Excel 的菜单栏。这背后的核心是 COM（组件对象模型）。微软通过操作系统底层的协议，让不同的可执行程序能够互相‘吞噬’和嵌套。虽然这成功建立起了 Office 帝国的格式霸权，但这种组件间的强耦合极其笨重，常常因为一个组件崩溃导致整个文档死锁，完美诠释了‘Andy (Intel) gives, Bill (Gates) takes away’的性能黑洞。
* **Borland 与 IBM 的反击：Object Windows Library 与 SOM/OpenDoc**
    作为微软当年的死敌，Borland 拥有极其优雅的面向对象框架（如 OWL），他们更倾向于在语言和框架层解决这种复用性。而 IBM 则联合 Apple、Borland 等公司推出了 OpenDoc 规范，底层依托于 SOM（系统对象模型）。他们的哲学是‘以文档为中心’，认为用户不应该关心自己是在用 Word 还是 Excel，所有的文字、表格、图表都应该是一个个独立的‘零件（Part）’，可以在一个统一的容器里自由嵌套。这种设计在架构上比 OLE 优雅得多，甚至计划跨平台推向 OS/2、Mac 和 Windows。
* **macOS 的演进：从 OpenDoc 的溃败到 Cocoa 的优雅架构**
    早期的 Apple 曾把赌注压在 OpenDoc 上，但随着乔布斯回归，Apple 果断砍掉了这个过于庞大复杂的计划。后来，macOS 依托于源自 NeXTSTEP 的 Cocoa 框架，走了一条截然不同的道路。Cocoa 完美地利用了 Objective-C 的动态特性，将视图层（View）抽象成了严格的树状层级结构（View Hierarchy）。在 macOS 看来，无论是文档、表格还是文本框，在底层都是一个 `NSView`。由于这种天然的自相似几何分形设计，嵌套变得极其自然，也为后来的富文本和排版引擎奠定了极为优雅的基础。
* **Linux 与开源阵营：Unix 哲学的挣扎与 KParts/Bonobo 的兴起**
    而在 Linux 领域，经典的 Unix 哲学宣扬‘做一件事并把它做好’，通过管道传输纯文本。但面对桌面办公的复合文档需求，这种纯文本哲学碰了壁。为了对抗微软，KDE 社区开发了 KParts，GNOME 社区开发了 Bonobo。Linux 阵营试图用轻量级的进程间通信（IPC）来让不同的独立程序（如电子表格组件和文本编辑器组件）在桌面容器里无缝嵌入。虽然这保留了开源的灵活性，但由于缺乏统一的商业标准去强推，导致格式兼容性和底层性能长期处于补丁叠补丁的挣扎状态。

### 3. 历史交锋：商业大合并与源代码长征

阿超看着小飞在白板上列出的这四条历史路线，赞许地笑了笑：“没错。当年的 Borland、IBM、Apple 和 Linux 社区，为了解决这个‘分形维度’的复杂度，在架构上付出了无数血汗。这里面分化出了两大工程流派：**架构契约派**与**物理死磕派**。”

小飞把白板中央的几条路线擦掉，重新梳理出一条清晰的商业与技术主线：

“行，那咱们就从这场上世纪 90 年代最惊心动魄的‘复仇者联盟’大合并说起。

当时，微软的 Office 战车已经隆隆开来。单打独斗的诸侯们为了活命，结成了‘三剑客同盟’：那时候电子表格的老大还是 Lotus 1-2-3，字处理的老大是 WordPerfect，而数据库和开发工具的老大则是大名鼎鼎的 Borland。为了对抗微软，他们把 **WordPerfect（字处理） + Borland Quattro Pro（电子表格） + Borland Paradox（数据库）** 强行打包在一起，冠以 **Borland Office** 的名字进行捆绑销售。

但这终究只是松散的商业联盟。到了 1994 年，网络巨头 **Novell** 为了打造能和微软全面抗衡的操作系统与办公生态，决定玩一把大的。他们狂砸巨资进行**大合并**——直接全资收购了 WordPerfect 公司，同时斥巨资买下了 Borland 的 Quattro Pro 团队和整条产品线，试图把它们熔炼成一个真正的铁拳产品。”

果冻听到这里，忍不住问：“既然大家都变成一家人了，代码合并、互相嵌套不是顺理成章的事吗？就像微软用 OLE 那样？”

“顺理成章？”小飞冷笑了一声，“微软用 OLE 实现‘套娃’，那是仗着 Windows 操作系统的底层特权，强行让不同的进程互相‘吞噬’。但 **Borland 的技术专家们骨子里极其高傲，他们根本不信这个邪！**”

小飞在白板上狠狠地画了一道横线：

“这帮顶尖的极客认为，微软的 OLE 协议极其臃肿、笨重，甚至是在出卖软件的控制权。他们心想：*‘凭什么我的软件里要塞进你微软的进程？凭什么我的架构要受你操作系统的摆布？为了追求极致的性能和跨平台控制力，我们就是要从源代码攻克这个微软用 OLE 解决的问题，在一个 App 中写出两个 App 的源代码！’*

于是，一场在今天看来近乎疯狂、也极其壮烈的‘源代码长征’爆发了。

Quattro Pro 的技术专家们不信邪，他们没有去调用外部的 WordPerfect 组件，而是**从零开始，在自己的电子表格应用内部，纯手工、像素级地把一个全功能的文字处理排版和渲染引擎重新写了一遍！**

他们甚至还顺手写了一套私有的、跨平台的异步数据同步框架叫 OBEX。他们的目标是：当用户在 Quattro Pro 的单元格里需要排版富文本、调整复杂的字体和段落时，不需要拉起任何外部进程，Quattro Pro 的内核自己就能轰鸣着把字处理功能跑起来。他们试图以这种‘物理死磕’的决绝，在自己的产品里硬生生用肉身重写、去满足所有套娃需求。”

果冻听得目瞪口呆，“这……这不就是拿肉身去抗分形需求的无限变异吗？结果呢？”

阿超叹了口气，把手里的记号笔扔回桌上：

“结果就是，他们遭到了复杂度的物理反噬，这也是**物理死磕派**的宿命。

这帮全宇宙最聪明、最硬核的工程师，把原本可以用来迭代核心业务的‘研发带宽’，全部无端消耗在了‘重复发明轮子’上。一个电子表格团队，竟然要分出核心精力去维护一个巨型字处理引擎。

当微软的**架构契约派**靠着生态胜利、通过 OLE/COM 组件契约实现‘高维度的 DRY 原则（不要重复你自己）’，并结合源源不断的用户反馈轻松搞定多维变异时，Novell 拼凑出的 PerfectOffice 却在底层‘代码巨兽’的内耗中溺水了。两大团队的代码底层完全不通，拼命重写的结果是软件体积急剧膨胀，Bug 满天飞，最终导致研发带宽断裂。

到了 1990 年代中期，面对 Windows 95 的洪流，研发带宽彻底崩盘的 Novell 终于认输，不得不把这两个难兄难弟再次打包，惨淡地转卖给了 Corel 公司。”

他转回身，目光重新落在果冻身上：

“果冻，你看到了吗？从 1980 年到 2000 年，这二十年桌面软件的战争史，本质上就是无数天才架构师在替用户收拾‘自相似递归需求’的战场。如果当年他们也像你写大作业一样，遇到新需求就硬编码一个新控件、复制一个新项目，那计算机行业早在四十年前就陷入焦土了。

你拿了九十五分的大作业惯性，在商业世界里连高维专家的肉身长征都扛不住。现在，你还觉得遇到多变的需求，靠简单的复制代码和硬编码，能撑得过产品的第一周期吗？”

### 4. 四十年后 Bug 出现

果冻原本还在惊叹于微软当年靠 OLE/COM 契约击败 Borland 的宏大叙事，正要把微软的架构师当成神来拜，小飞突然冷笑了一声，把自己的 Windows 11 笔记本转了过来，屏幕上正卡着一个死活点不掉的系统弹窗：

`⚠️ 提示：Microsoft Excel 正在等待其他某个应用程序完成对象链接与嵌入操作。`

![简单的操作后，出现了奇怪的 OLE 错误对话框](image-1.png)

小飞指着这个弹窗说：“兄弟，看。经历了四十年的风风雨雨，硬件主频翻了成千上万倍，操作系统都迭代到 Windows 11 了。Word 和 Excel 都有 CoPilot 的小按钮在一旁伺候，但是，今天在 Word 里双击一个嵌入的 Excel 图表，这个当年 OLE 时代留下的‘远古幽灵’很有可能跳出来，把你的 Word 和 Excel 应用卡死。”

果冻揉了揉眼睛，满脸不可思议：“这都 2026 年了！微软那么多天才工程师，怎么连这个弹窗都干不掉？！”

果冻尝试了“确定”和“取消”，也退出了 Word 和 Excel 程序，但是不论怎么操作，大约一分钟后，这个对话框又静静地出现在屏幕上了。

“为什么一个普通的办公软件之间的交流问题，退出办公软件都不能解决？难道 OLE/COM 根本就不是 Office 软件的功能，而是 Windows 的系统级组件吗？”

“你猜得完全没错，”阿超从座位上站起来，在白板上把原先画在 Word 和 Excel 底下的那条基座线狠狠地加粗，“COM 根本不是 Office 的附庸，它就是 Windows 操作系统的骨架。”

阿超在白板上画出了一幅清晰的底层架构图：


```

+-------------------------------------------------------+
|  应用层：    Word 进程   <======>   Excel 进程        |
+-------------------------------------------------------+
|  组件层：             O L E  协 议                     |
+-------------------------------------------------------+
|  内核层：  COM 运行时环境 (Ole32.dll / RPCRT4.dll)    |
|            - 全局类厂注册表 (ROT)                      |
|            - 跨进程通信守护 (RPC / LPC)               |
+-------------------------------------------------------+
|  操作系统：           Windows 11 内核                  |
+-------------------------------------------------------+

```

“当年微软为了让 OLE 拥有绝对顶级的性能，并没有把它写成一个应用层的动态链接库（DLL），而是直接把它熔铸进了 Windows 里。

我们在 Windows 11 里调用 OLE，底层支撑它的是像 `Ole32.dll` 和 `RPCRT4.dll` 这样的核心系统组件。当你双击嵌入对象时，操作系统级别的 RPC（远程过程调用）引擎和全局运行对象表（ROT, Running Object Table）就会全权接管这两个应用的生命周期。”

小飞在旁边补充，“这就带来了一个极其致命的物理后果：资源的全局宿主化。”

“对，”阿超转过身，“正常情况下，如果一个普通软件卡死了，你打开任务管理器，‘结束任务’强行杀掉它的进程就行了。因为现代操作系统的进程安全隔离机制，会自动帮你回收这个进程申请的所有内存和句柄（Handle）。

但在 OLE 复杂的分布式死锁里，情况完全失控了：

* **系统级服务被扣为人质**：Word 和 Excel 的跨进程通信，是通过内核态的 COM 运行时环境进行转接的。当 Excel 崩溃或由于底层锁争用（Lock Contention）导致 COM 管道彻底锁死时，它不仅卡死了自己，还把操作系统里负责调度跨进程通信的全局 RPC 线程池给占死或者带崩了。
* **句柄与注册表死锁**：在 OLE 握手的状态下，Word 在内核中持有着 Excel 的进程句柄，操作系统将它们视为一个‘复合原子操作’。当你试图强行杀死 Excel 时，内核服务会因为 Word 的主线程依然处于‘等待挂起’状态，而拒绝强行释放底层的安全句柄。
* **任务管理器失效**：你有时会发现，在 OLE 死锁发生时，你甚至连‘任务管理器’都呼不出来，或者在任务管理器里点击‘结束任务’，那个卡死的进程依然顽固地躺在那里，CPU 占用率高居不下。这是因为底层的内核线程已经陷入了不可中断的等待状态（Uninterruptible Sleep），操作系统内核为了防止系统崩溃，宁可让这个进程变成‘僵尸’，也不敢强制回收它。”

大家各种咨询，问了几家 AI 助手，终于在 Windows PowerShell 中运行下面的命令把恼人的对话框消除了。

#### 1. 禁用 Office 核心的定时维护和自动拉起任务
```powershell
Disable-ScheduledTask -TaskName "Office Automatic Updates 2.0" -ErrorAction SilentlyContinue
Disable-ScheduledTask -TaskName "Office Feature Updates" -ErrorAction SilentlyContinue
Disable-ScheduledTask -TaskName "Office Feature Updates Log Collection" -ErrorAction SilentlyContinue

```
