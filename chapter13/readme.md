## 13.1 由核事故引出的基本名词解释

1986年切尔诺贝利核事故中，辐射检测仪指针卡死在最大量程3.6伦琴/时，管理人员误判为“可控”。这一事件深刻揭示了软件测试的核心：我们不仅要发现目标系统的Bug，更要确保监测工具本身可靠。测试的根本目标是回答V&V两个问题——Verification（我们是否正确地构建产品？）与Validation（我们是否构建了正确的产品？）。在AI时代，这一目标被扩展为同时验证代码逻辑、模型行为、数据质量与服务稳定性。

## 13.1 “伙伴测试”的非正式性会不会削弱其在敏捷开发中的价值？

 学生认为伙伴测试非正式，依赖自觉性，进度紧张时易走过场，询问企业如何保证效果，是否需要检查清单。

**问题的核心：** 非正式 ≠ 无效。有效性靠责任绑定和自动化门禁兜底，而非流程正式化。

**回答：**

伙伴测试的非正式性是优点——低门槛、快速反馈。企业保证效果的方法：
1. **最低标准**：PR必须标注“Buddy Test Passed”才能合并。
2. **自动化兜底**：CI门禁（单元测试、覆盖率）会拦住真正破坏，伙伴测试只负责快速发现明显问题。
3. **责任绑定**：签入后出现严重Bug，复盘伙伴测试环节。

**扩展思考：**

书里13.3.2节提到的“智能测试平台”让AI充当虚拟伙伴，7x24小时待命。非正式性保留，可靠性由机器保证。

## 13.2 “探索式”的测试应当主要放在软件开发和测试的什么阶段？

 书中提到“探索式测试太多是管理不佳”，学生询问其最佳实施阶段。

**问题的核心：** 探索式测试不是随机乱测，而是有目标、有时间限制的智力活动。最佳时机是自动化测试覆盖后、发布前的窗口期。

**回答：**

**最佳阶段：**
- 新功能稳定后，自动化测试已通过，做探索式测试找边界和体验问题。
- 发布前冲刺，集中1-2天专门做。
- 修复复杂Bug后，顺便探索周边功能。

**什么时候不该做？** 项目初期、自动化测试未跑通时。那时做探索式测试确实说明管理不佳。

**扩展思考：**

探索式测试是发现“纸割伤”（12.3.3节）的利器。自动化测不到“按钮藏太深”“每次重新选字体”这类问题，活人随便点几下就能发现。

## 13.3 单元测试应该在什么时候开始做？

 询问单元测试应在代码完成前还是后设计，测试标准应该是设计的功能还是程序实际的功能。

**问题的核心：** 单元测试应在写代码之前设计（TDD），标准是预期功能，不是代码实际做的。

**回答：**

正确顺序：
1. 写接口定义
2. 写单元测试（定义输入、预期输出）
3. 写代码让测试通过
4. 重构

测试标准是**预期功能**。需求变化时，先改测试，再改代码。不要根据写完的代码写测试——那是验证，不是设计。

**扩展思考：**

AI时代，先写测试更重要。AI可根据测试生成正确代码。测试是人与AI之间的契约。

## 13.4 伙伴测试能否用结对编程代替？

 询问伙伴测试能否用结对编程代替，并猜测结对编程成本更高。

**问题的核心：** 两者目的不同——结对编程是预防，伙伴测试是验证。不能互相代替。

**回答：**

| 维度 | 结对编程 | 伙伴测试 |
|------|----------|----------|
| 时间投入 | 两人全程 | 签入前快速验证 |
| 成本 | 高 | 低 |
| 目的 | 预防缺陷 | 发现缺陷 |

当测试人员远少于开发人员时，让测试人员结对编程会导致项目停摆。伙伴测试是轻量级替代方案。

**扩展思考：**

书里13.3.2节的“智能测试平台”让AI做快速验证，人类测试人员可同时与多个开发者异步协作，成本更低、覆盖更广。

## 13.5 如何为有副作用的方法设计完备的测试用例？

 学生发现与外部交互（数据库、控制台）的方法难以测试，常跳过，但Bug多出在此，询问如何设计测试。

**问题的核心：** 不要死磕单元测试。用Mock隔离，或用集成测试覆盖副作用。

**回答：**

**解决方案：**
1. **依赖注入 + Mock**：不连真实数据库，Mock返回固定数据。需重构代码，把副作用推到边界。
2. **集成测试**：用Docker起测试数据库，每次测试前重置。成本高，但对核心路径值得做。
3. **契约测试**：对第三方API，验证返回格式是否符合预期。

**策略**：单元测试测纯逻辑，集成测试测协作。不要用集成测试替代所有单元测试。

**扩展思考：**

用FMEA（13.4.2节）评估副作用严重程度。对支付、删除等高危模块，多做真实集成测试；对日志等低风险副作用，大胆Mock。

## 13.6 关于AI批改功能的沙盘推演问题

**学生提问简述（沙盘推演）：** AI批改与黄金数据集吻合度99%（验证通过），但30%学生点击申诉。问是否是Bug，测试错在哪里。

**问题的核心：** 验证通过不等于确认通过。学生申诉是用户不信任问题，不是功能正确性问题。

**回答：**

这不是传统Bug，而是**价值确认失败**。原因可能：
- 黄金数据集与学生认知不符
- AI可解释性差（只给分数不给具体建议）
- 学生期望被理解，不只是要分数

**测试错在哪里？** 只做了功能验证（Verification），没做用户验收测试（Validation）中的感知质量测试。

**扩展思考：**

书里13.6节的“三层测试框架”中，第三层“业务验收”要求真实场景验证。上线前应找学生试用，观察申诉率。AI批改不是数学题，涉及人的自尊。**测试的终极目的不是“证明没有Bug”，而是“让用户在真实世界里安心使用”。**
